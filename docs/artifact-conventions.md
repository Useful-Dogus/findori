# 에이전트 산출물 작성 규칙

에이전트 종류와 무관하게 적용되는 단일 산출물 작성 규칙.
이슈·커밋·PR 형식, 번호 정렬 체계, 코드 품질 기준, hotfix 예외 흐름을 정의한다.

> **스펙 워크플로우**: 이 프로젝트는 speckit을 기본 워크플로우 도구로 사용한다. 파일 구조·단계 순서는 speckit 도구 정의를 따른다.

> **소급 적용 없음**: 기존 머지된 커밋·PR은 이 규칙의 적용 대상이 아니다.
> 새로 생성되는 산출물에만 적용된다.

---

## 1. 번호 정렬 체계

GitHub 이슈 번호, 브랜치 번호, 스펙 디렉터리 번호는 항상 일치해야 한다.

### 규칙

| 항목 | 형식 | 예시 |
|------|------|------|
| GitHub 이슈 | `#N` (GitHub 자동 부여) | `#42` |
| 브랜치 | `NNN-<short-name>` (3자리 제로패딩) | `042-agent-artifact-conventions` |
| 스펙 디렉터리 | `specs/NNN-<short-name>/` (3자리 제로패딩) | `specs/042-agent-artifact-conventions/` |

**패딩 규칙**: 브랜치명·스펙 디렉터리에는 `printf "%03d"` 형식으로 3자리 제로패딩을 적용한다.
100 이상 번호는 자연수 그대로 사용한다.

**이슈 참조 시 패딩 없음**: 커밋 메시지(`[Issue #N]`)와 PR 본문(`Closes #N`)에서는 원래 숫자를 그대로 사용한다.

```
GitHub #42  →  브랜치: 042-feature-name  →  specs/042-feature-name/
커밋: [Issue #42] ...   PR: Closes #42   (패딩 없음)
```

---

## 2. GitHub 이슈 형식

### 제목

자유 형식. GitHub이 이슈 번호를 자동 부여한다.

### 필수 섹션

모든 이슈에 아래 3개 섹션이 있어야 한다.

```markdown
## 배경
[이 이슈가 왜 필요한지 — 문제, 맥락, 동기]

## 범위

**In Scope**
- [포함되는 항목]

**Out of Scope**
- [제외되는 항목]

## 완료 기준 (DoD)
- [ ] [검증 가능한 완료 조건]
```

### 선택 섹션

```markdown
## 의존성
- [선행 이슈 또는 외부 조건]
```

## 3. 커밋 메시지 규칙

### Tier 1 — 이슈 구현 주 커밋

이슈 기반 피처의 주 커밋(브랜치 주요 구현 완료, squash-merge 제목)에 사용한다.

```
[Issue #N] <설명>
```

예시:
```
[Issue #42] 에이전트 산출물 작성 규칙 통합
```

### Tier 2 — 보조·수정 커밋 (Conventional Commits)

이슈 없는 수정, hotfix, 소규모 변경에 사용한다.

```
<type>[(<scope>)]: <설명>
```

**허용 타입 집합**:

| 타입 | 용도 |
|------|------|
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `hotfix` | 긴급 프로덕션 버그 수정 |
| `docs` | 문서만 변경 |
| `chore` | 설정·도구·스크립트 변경 (프로덕션 무관) |
| `refactor` | 동작 변경 없는 코드 구조 개선 |
| `test` | 테스트 추가·수정 |
| `style` | 코드 포매팅 등 로직 무관 변경 |
| `perf` | 성능 개선 |
| `ci` | CI/CD 설정 변경 |

스코프는 선택 사항이다: `fix(auth): ...`, `chore(deps): ...`

### 금지 항목

- AI 도구명 기반 `Co-Authored-By` 라인이나 특정 에이전트 협업자 표기 금지
- 허용 타입 집합 외 타입 사용 금지

## 4. PR 형식

### 제목

```
[Issue #N] <설명>
```

Tier 1 커밋 메시지와 동일 형식을 사용한다.

### 필수 섹션

```markdown
## Summary
- [변경 내용 핵심 요약 (불릿 포인트)]

## Test plan
- [x] 실행한 검증 항목
- [x] `npm run validate` 통과
- [x] `npm run test` 통과 (해당 시)
- [x] `npm run build` 통과 (배포 대상 변경 시 필수, Next.js 라우트/빌드 검증 포함)

Closes #N
```

기본 원칙: 웹 애플리케이션 코드 변경 PR은 `npm run validate`, `npm run test`, `npm run build`를 모두 실행하고 Test plan에 명시한다.

### 선택 섹션

```markdown
## Changes
[상세 변경 목록. 복잡한 구현에서 권장]
```

### 링크 키워드 규칙

| 키워드 | 사용 조건 |
|--------|-----------|
| `Closes #N` | 이 PR이 해당 이슈를 완전히 닫는다 |
| `Part of #N` | 이슈의 일부만 구현한다 |
| `Refs #N` | 이슈를 닫지 않고 참조만 한다 |

hotfix 등 이슈가 없는 PR은 링크 키워드를 생략할 수 있다.

## 5. 코드 품질 기준

에이전트 종류와 무관하게 코드 생성·수정 시 아래 4개 축을 기준으로 삼는다.

### 가독성

- **단일 책임**: 함수·모듈은 하나의 목적만 수행한다.
- **의도가 드러나는 이름**: 변수·함수·타입 이름만으로 역할을 알 수 있어야 한다. 약어·범용 이름(`data`, `tmp`, `x`) 지양.
- **매직 넘버·문자열 금지**: 리터럴 값은 이름 있는 상수로 추출한다.
- **짧고 집중된 함수**: 화면 한 페이지를 넘기 전에 기능 단위로 분리를 고려한다.

### 예측 가능성

- **프로젝트 관용구 우선**: 기존 코드베이스의 패턴·컨벤션을 새 코드에도 동일하게 적용한다. 불필요한 스타일 혼합을 피한다.
- **사이드이펙트 최소화**: 함수가 인자 외 외부 상태를 변경한다면 이름·주석으로 명시한다.
- **암묵적 타입 변환 금지**: TypeScript 환경에서 `as any`, 암묵적 `undefined` 반환을 피하고 명시적 타입을 사용한다.

### 응집도

- **관련 로직 한 곳에**: 동일 도메인의 로직을 여러 파일에 분산하지 않는다.
- **불필요한 추상화 지연**: 중복이 세 번 이상 발생하기 전까지 공통 유틸리티 추출을 서두르지 않는다.
- **레이어 역할 준수**: 라우트 핸들러는 비즈니스 로직을 갖지 않고, 서비스는 HTTP 컨텍스트를 알지 않는다.

### 결합도 & 기본 성능

- **의존성 단방향**: 모듈 의존성 방향을 단방향으로 유지한다. 순환 의존성을 도입하지 않는다.
- **N+1 쿼리 금지**: 루프 내 DB 쿼리는 배치 쿼리 또는 join으로 대체한다.
- **무한 루프·메모리 누수 주의**: 종료 조건 없는 재귀·루프, 해제되지 않는 이벤트 리스너·구독을 만들지 않는다.
- **불필요한 렌더링 방지**: React 컴포넌트에서 매 렌더마다 새 객체·함수를 생성하는 패턴을 피한다.

## 6. hotfix 예외 흐름

### hotfix 정의

spec/plan/tasks 사이클 없이 즉각 수정이 필요한 긴급 버그. 프로덕션 또는 main 브랜치에서 즉시 대응이 필요한 경우.

### 커밋

- 타입: `hotfix: <설명>` 또는 `fix: <설명>`
- 이슈가 있으면 `Refs #N` 포함 권장

### PR 최소 필수 항목

```markdown
## Summary
- [최소 한 줄 이상]

## Test plan
- [최소 한 줄 이상]
```

가능하면 hotfix도 `npm run build`까지 검증한다. 빌드가 현실적으로 불가능한 경우에만 생략 사유를 Test plan에 적는다.

### 생략 가능 항목

- spec.md, plan.md, tasks.md
- `Closes #N` (이슈가 없을 경우)
