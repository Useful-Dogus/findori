## 1. 현재 파이프라인 진단: 왜 기계적인가?

### 1-1. Temperature 0.2 — 가장 즉각적인 원인

```ts
temperature: 0.2
```

구조적 정확성(JSON 출력)은 `tool_use` 스키마가 보장하므로, 언어 표현에까지 이 제약을 걸 필요가 없다. **0.7로 올리는 것만으로도 체감 차이가 크다.**

### 1-2. 시스템 프롬프트가 "규칙 준수 기계"를 만들고 있다

현재 시스템 프롬프트 구성비:
- 카드 타입 형식 규칙: ~60%
- 금지 조항 (7회): ~15%
- 내러티브 흐름 지침: ~15%
- **독자가 흥미를 느끼게 하는 방법**: **0%**

### 1-3. 카드 타입 시스템이 너무 범용적이다 ← **[R4]에서 새롭게 발견**

현재 7개 카드 타입(cover/reason/bullish/bearish/community/stats/source)은 모두 `title + body`라는 동일한 필드 구조를 공유한다. LLM은 항상 **"제목 + 3줄 설명" 패턴을 반복**한다. 독자는 두 번째 카드부터 패턴을 예측하고 스와이프를 멈춘다.

```typescript
// ❌ 현재 — 모든 카드가 같은 구조
{ type: "reason", tag: string, title: string, body: string, visual: ... }
{ type: "bullish", tag: string, title: string, body: string, visual: ... }
{ type: "bearish", tag: string, title: string, body: string, visual: ... }
```

카드 타입이 달라도 필드가 같으면 → 내용도 같은 패턴이 된다.

### 1-4. 기사 content 500자 — 재료가 부실하다

한국어 500자는 본문 2~3 단락 수준. stats 카드에 수치가 없고, reason 카드가 "이런 이유로 상승했다" 수준에 그친다.

### 1-5. 필터에 "흥미도" 기준 없음

투자 관련성이 높아도 재미없는 기사가 있다. "독자가 10초 안에 계속 읽고 싶어지는 기사"라는 기준이 없다.

## 2. 과학적 기반: 심리 메커니즘 [R3]

### 도파민은 쾌감이 아닌 예측오류 신호다

**보상예측오류(Reward Prediction Error)**: 기대를 깨는 순간에 도파민이 반응한다. 절대값보다 **변화량(Δ)** 이 더 강한 반응을 만드는 이유.

### 세 가지 핵심 심리 메커니즘

**① 정보 격차 이론 (Loewenstein)**: 호기심 = "지금 아는 것"과 "알고 싶은 것"의 격차. 커버에서 격차를 의도적으로 만들고, 카드를 넘길수록 메워주는 구조.

**② 처리 유창성**: 쉽게 읽힐수록 좋다는 느낌이 든다. 높은 대비, 예측 가능한 레이아웃, 짧은 텍스트 = 신뢰 향상.

**③ 목표 경사 효과**: 목표에 가까울수록 행동 속도가 빨라진다. "7장 중 3장" 같은 진행도 표시는 완주율을 높인다.

### Zeigarnik 효과는 과학적 근거가 약하다

"결론 숨기기"는 최근 메타분석에서 보편적 효과가 없다고 나왔다. 투자 콘텐츠에서는 오히려 신뢰와 법적 리스크를 해친다. **권장하지 않는다.**

### 윤리적 경계선

"이해를 통한 통제감"을 주는 설계이지, "강박적 새로고침"이나 "불안/공포 기반의 충동 클릭"을 만드는 설계가 아니다.

## 4. 좋은 콘텐츠의 핵심 원칙

### 4-1. 스와이프 이유 4가지 [R3, R4]

1. **새로움/Δ**: 오늘 달라진 것 (절대값 < 변화량)
2. **보상 예고**: 한 장 넘기면 얻는 것
3. **정보 격차**: "이게 왜?", "진짜로?"
4. **자기 관련성**: 내 포지션, 내 지갑과의 연결

### 4-2. 커버 훅 공식 [R4]

| 공식 | 예시 |
|------|------|
| `[기간] 만에 [수치]` | "2년 만에 1,500원 — 2009년 이후 처음" |
| `[현상]인데 왜 [의외 결과]?` | "역대 최대 실적인데 주가는 -9%, 왜?" |
| `[수치]가 내 [자산]에 미치는 영향` | "내 해외직구 비용, 얼마나 올랐을까요?" |
| 상식 반박형 | "달러 인덱스는 정상인데, 원화만 혼자 빠졌어요" |

금지 패턴:
- "삼성전자 주가 상승" → 수치 없음
- "코스피 하락세" → 얼마나? 왜? 없음
- "OO 기업 실적 발표" → 결과 없음

### 4-3. 결과 → 원인 순서 [R4]

```
❌ "이란이 호르무즈 해협을 위협해서... 에너지 가격이... 원화가 떨어졌어요."
✅ "외국인 자금 1.8조원이 하루 만에 빠졌어요. 이란이 원유 수송로를 막겠다고 위협했거든요."
```

### 4-4. 절대값보다 변화량(Δ) [R3, R4]

```
❌ "SK하이닉스 영업이익 47.2조원"
✅ "1년 만에 +101% — 반도체 역사상 최고 수익성"
```

### 4-5. 한 장 = 메시지 1개 [R2, R3, R4]

한 장 = 메시지 1개 = 데이터 1개 = 해석 1개. 텍스트는 3줄 이내. 편집이 안 되면 카드를 둘로 나눌 것.

### 4-6. 콘텐츠 타입별 2:1 비율 [R3]

속보형 (Δ) 2 : 교육형 (구조) 1. 속보형은 즉시성과 새로움을, 교육형은 저장·재방문을 유도한다.

### 4-7. 말투 안티패턴 [R4] ← **신규**

LLM 생성 티가 나는 특정 패턴들:

```
❌ "~가 아니다. 대신 ~다." 패턴 → LLM 티
❌ "즉," "결국," 접속사 남발 → 독자가 눈치챔
❌ 어미 혼용 ("~했어요"와 "~했다" 섞기) → 어색함
❌ 전문용어 첫 등장 시 설명 없음 → 이탈 유발
```

어미 체계는 "~했어요/이에요" 또는 "~했다/이다" 중 하나로 통일. 전문용어는 첫 등장 시 괄호 설명 필수.

### 4-8. 신뢰성 = 비교 데이터 [R4]

단독 주장은 신뢰를 얻기 어렵다. 비교 데이터를 함께 제시할 것.

```
❌ "원화만 유독 약세"
✅ 유로·파운드 +10% 강세와 원화를 나란히 비교
```

## 5. 카드 타입 시스템 재설계 [R4] ← **핵심 신규 섹션**

### 왜 카드 타입을 바꿔야 하는가

현재 구조의 근본 문제: 모든 카드가 `title + body`라는 같은 필드를 가진다 → LLM이 항상 같은 패턴을 쓴다 → 독자가 두 번째 카드부터 스와이프를 멈춘다.

**카드 타입마다 완전히 다른 필드 구조를 정의하면** 필드가 다르므로 내용 패턴도 자연히 다르다.

### 새로운 카드 타입 정의

```typescript
// 극적인 수치 변화를 시각화
type DeltaCard = {
  type: "delta"
  before: string      // "1,280원"
  after: string       // "1,500원"
  period: string      // "2년 만에"
  context: string     // 1문장 해석 (3줄 이내)
}

// 처음 등장하는 주체 소개 + 변화
type DeltaIntroCard = {
  type: "delta-intro"
  before: string
  after: string
  period: string
  what: string        // "ARM Holdings"
  whatDesc: string    // 1~2문장 설명
  trigger: string     // 이슈 발생 이유 1문장
}

// 결과 먼저, 원인 파고들기
type CauseCard = {
  type: "cause"
  result: string      // 결과 (짧게, 강조 표시됨)
  cause: string       // 원인 설명 (3줄 이내)
}

// 충격적인 단일 통계 + 상식 뒤집기
type StatCard = {
  type: "stat"
  number: string      // "771조원"
  label: string       // "국민연금 해외 투자 규모"
  reveal: string      // 상식을 뒤집는 해석 (2줄)
}

// 비교 테이블 — 신뢰 구축
type CompareCard = {
  type: "compare"
  q: string
  rows: {
    label: string
    value: string
    dir: "up" | "down" | "worst"
    note: string
  }[]
  footer: string
}

// 독자 지갑 직접 연결
type ImpactCard = {
  type: "impact"
  items: {
    label: string
    before: string
    after: string
    diff: string      // "+22,000원" 또는 "위험" / "주의"
  }[]
}

// 마지막 카드 — 결론
type VerdictCard = {
  type: "verdict"
  verdict: string     // 한 문장 결론
  reasons: string[]   // 근거 3개 이내
}

// 다음 카드로 넘기는 서스펜스
type QuestionCard = {
  type: "question"
  q: string
  hint: string        // 답의 방향을 암시
}
```

### 타입 선택 가이드

| 상황 | 추천 타입 |
|------|----------|
| 숫자가 드라마틱하게 변했을 때 | `delta` |
| 처음 등장하는 종목/주체 소개 | `delta-intro` |
| 독자가 결과는 알지만 이유를 모를 때 | `cause` |
| 충격적인 단일 통계가 있을 때 | `stat` |
| 비교 데이터로 신뢰를 더해야 할 때 | `compare` |
| 독자 실생활과 연결할 때 | `impact` |
| 이슈 마지막 장 | `verdict` |
| 다음 카드로 넘기는 흥미 유발 | `question` |

### 현재 타입 → 새 타입 매핑

| 현재 타입 | 새 타입 | 변경 이유 |
|---------|--------|---------|
| cover | `delta` 또는 `delta-intro` | 커버 유형을 이슈에 따라 선택 |
| reason | `cause` | 결과→원인 순서 강제 |
| bullish | `cause` 또는 `stat` | 상승 근거를 구체적 수치로 |
| bearish | `cause` 또는 `compare` | 리스크를 비교로 맥락화 |
| community | `question` 또는 `compare` | 가짜 커뮤니티 반응 제거 |
| stats | `stat` (단일) + `compare` (비교) | 목적에 따라 분리 |
| source | 유지 | |

### 권장 카드 시퀀스

```
1. 커버:      delta 또는 delta-intro  (Δ 수치 + 정보 격차 훅)
2. 1~2장:    cause, stat              (결과 먼저 → 원인 파고들기)
3. 3~4장:    cause, stat, compare     (근거 강화)
4. 5장:      impact                   (독자 생활 연결)
5. 마지막:   verdict                  (한 문장 결론)
```

## 6. 파이프라인 재설계 방향 [R4]

### 현재 구조 vs 권장 구조

```
현재: collect → filter(Haiku) → generate(Sonnet) → store

권장: collect → filter(Haiku) → fact_extract(Haiku) → generate(Sonnet) → store
```

### 2단계 LLM 호출의 장점

**1단계: 팩트 추출 (Haiku)**
- 입력: 기사 원문 (길고 비구조화)
- 출력: 핵심 수치, Δ 값, 주요 인물/기관, 원인-결과 관계 (~200 토큰)
- 결과물 예시:

```json
{
  "entity": "ARM Holdings",
  "delta": { "before": "$134", "after": "$162", "period": "3월 25일 하루" },
  "cause": "처음으로 직접 칩 설계 발표 (ARM AGI CPU)",
  "key_stats": ["5년 내 목표 매출 $150억", "메타·OpenAI 채택"],
  "risks": ["현재 PER 182배", "단기 급등 후 조정 가능성"]
}
```

**2단계: 카드 생성 (Sonnet)**
- 입력: 압축된 팩트 (~200 토큰) + 카드 타입 스키마
- 출력: FEEDS JSON

**부가 효과:**
- 1단계 팩트를 캐싱해서 재생성 시 재사용 가능
- 각 단계가 독립 모듈 → 한 단계 실패해도 재시도 가능
- Haiku는 $0.80/MTok으로 저렴 → 팩트 추출에 최적

### 주제 선정 프롬프트 개선 [R4]

현재 Haiku 필터 기준에 추가할 내용:

```
다음 기사 목록에서 카드뉴스화할 주제를 3개 선정하세요.

선정 기준:
1. 구체적인 수치(Δ)가 있는 기사 우선
2. 25~40세 직장인 투자자가 "이거 친구한테 말해야겠다"고 느낄 주제
3. 오늘만 유효한 뉴스보다 1~2주 유효한 주제 선호
4. 독자의 지갑이나 포지션에 직접 영향을 주는 주제 우선

출력: { "topics": [{ "title": "", "reason": "", "key_facts": ["", ""] }] }
```

## 7. 이미지 전략 재설계 [R4]

### 현재 방식의 문제

```typescript
// 현재: 자유형 검색어 → 불일치 이미지, 토큰 낭비
{ imgQuery: "global financial crisis dollar exchange rate" }
```

### 권장 방식: 카테고리 키 풀

LLM이 긴 검색어 대신 짧은 카테고리 키만 출력한다.

```typescript
const IMAGE_POOL = {
  "stock-up":     ["url1", "url2", "url3"],
  "stock-down":   ["url1", "url2", "url3"],
  "currency":     ["url1", "url2", "url3"],
  "ai-chip":      ["url1", "url2", "url3"],
  "earnings":     ["url1", "url2", "url3"],
  "warning":      ["url1", "url2", "url3"],
  "growth":       ["url1", "url2", "url3"],
  "consumer":     ["url1", "url2", "url3"],
  "global-trade": ["url1", "url2", "url3"],
}

// 카드 JSON에서
{ "type": "stat", "imgCategory": "ai-chip", ... }
```

**장점**: 토큰 절약, 큐레이션 품질 보장, CDN 캐싱 가능, 브랜드 일관성 유지.

## 8. 시스템 프롬프트 개선 전략

### 핵심 전환: 규칙집 → 편집 가이드

```
현재: 형식 규칙 60% + 금지 조항 15% + 내러티브 흐름 15% + 스타일 0%
목표: 편집 페르소나 15% + 스타일+예시 40% + 구조 가이드 30% + 핵심 규칙 15%
```

### 반드시 추가해야 할 요소

**① 편집 페르소나**
```
당신은 "핀도리 데일리"의 수석 편집장입니다.
독자: 25~40세 직장인 투자자, 출퇴근 중 10분, 전업 투자자 아님
목표: "오, 이거 진짜 쓸만한데?" + 친구에게 공유하고 싶은 카드
참고: Morning Brew의 친근함 + 토스증권의 시각 감각 + 주식갤러리의 현실감
```

**② Δ 우선 원칙**

**③ 텍스트 3줄 이내 원칙**

**④ 말투 안티패턴 명시** (신규)

**⑤ 금지 조항 → 긍정형 전환**

**⑥ Few-shot 예시 추가** — 타입당 1개가 최적. 많을수록 토큰이 늘고 예시를 그대로 모방하는 부작용이 생긴다. [R4]

## 12. 성과 측정

### 공개 vs 내부 지표

공개 (소셜 블레이드): 팔로워, 추정 참여율, 평균 좋아요/댓글
내부 인사이트 필요 (진짜 중요한 지표):

| 지표 | 의미 | 개선 방향 |
|------|------|---------|
| **저장 수** | 재소비 가치 | stat·교육형 카드 강화 |
| **공유 수** | 확산력 | "친구에게 전달할 인사이트" 1개 |
| **완료율** | 서사 완성도 | 어느 카드에서 이탈하는지 파악 |

## 14. 단계별 실행 로드맵

현재 카드 타입 스키마를 유지하면서 프롬프트를 개선. 스키마 변경 없이 할 수 있는 최대치.

5. 금지 조항 → 긍정형 전환
6. 커버 훅 공식 4가지 + 예시 강화
7. community quotes 가이드 강화
8. stats change 필드 맥락 원칙 (절대값 → Δ+맥락)
9. **말투 안티패턴 명시** ("~가 아니다. 대신 ~다." 등) ← 신규

스키마 변경 + 파이프라인 재설계. DB 마이그레이션 필요.
