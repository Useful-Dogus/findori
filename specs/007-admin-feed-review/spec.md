# Feature Specification: Admin 피드 목록/날짜별 이슈 검토 화면 구현

**Feature Branch**: `007-admin-feed-review`
**Created**: 2026-03-03
**Status**: Draft
**Input**: User description: "#7 Admin 피드 목록/날짜별 이슈 검토 화면 구현"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 운영자가 날짜별 피드 상태를 한눈에 파악한다 (Priority: P1)

운영자는 `/admin` 진입 시 날짜별 피드 목록을 확인할 수 있어야 한다. 각 항목에서 피드 상태(`draft` / `published`)를 즉시 식별하고, 원하는 날짜의 이슈 검토 화면으로 이동할 수 있어야 한다.

**Why this priority**: 운영자가 어느 날짜의 피드를 처리해야 하는지 가장 먼저 파악해야 한다. 이 화면이 없으면 날짜별 이슈 검토, 승인, 발행 흐름 전체가 시작될 수 없다.

**Independent Test**: 로그인한 운영자가 `/admin`에서 날짜 목록과 각 상태를 확인하고, 특정 날짜를 클릭해 이슈 목록 화면으로 이동하면 통과.

**Acceptance Scenarios**:

1. **Given** 운영자가 인증된 상태로 `/admin`에 진입하면, **When** 피드 목록이 로드되면, **Then** 날짜별 피드 항목이 최신 날짜 우선으로 정렬되어 표시된다.
2. **Given** 특정 날짜의 피드가 `draft` 상태일 때, **When** 목록에서 해당 항목을 확인하면, **Then** `draft` 상태 표시가 노출된다.
3. **Given** 특정 날짜의 피드가 `published` 상태일 때, **When** 목록에서 해당 항목을 확인하면, **Then** `published` 상태 표시와 발행 시각이 노출된다.
4. **Given** 피드 목록에서 특정 날짜 항목을 클릭하면, **When** 이동이 완료되면, **Then** `/admin/feed/[date]` 화면으로 진입한다.
5. **Given** 아직 피드가 생성된 적 없는 초기 상태일 때, **When** 운영자가 목록을 확인하면, **Then** 빈 상태 안내 메시지가 표시된다.

### User Story 2 - 운영자가 특정 날짜의 이슈 목록과 카드 미리보기를 확인한다 (Priority: P1)

운영자는 `/admin/feed/[date]` 화면에서 해당 날짜에 생성된 이슈 목록을 확인하고, 각 이슈의 카드 내용을 미리볼 수 있어야 한다.

**Why this priority**: 이슈 목록과 카드 내용 확인이 가능해야 #8의 편집·승인·반려 기능을 의미 있게 연결할 수 있다. 목록만 보여도 "어떤 이슈가 생성됐는가"를 파악하는 독립적 가치가 있다.

**Independent Test**: 로그인한 운영자가 `/admin/feed/2026-03-03` 같은 주소로 접근했을 때 이슈 목록이 표시되고, 이슈를 선택하면 카드 내용(제목, 태그, 카드 장 수)을 확인할 수 있으면 통과.

**Acceptance Scenarios**:

1. **Given** 특정 날짜에 이슈가 생성되어 있을 때, **When** 운영자가 `/admin/feed/[date]`에 진입하면, **Then** 해당 날짜의 이슈 목록이 표시된다.
2. **Given** 이슈 목록이 표시된 상태에서, **When** 운영자가 이슈 항목을 확인하면, **Then** 이슈 제목, 엔티티 이름, 상태, 카드 장 수가 노출된다.
3. **Given** 이슈 항목을 선택하면, **When** 카드 미리보기가 펼쳐지면, **Then** 각 카드의 타입, 제목, 핵심 텍스트가 표시된다.
4. **Given** 해당 날짜에 이슈가 없을 때, **When** 운영자가 해당 날짜 화면에 진입하면, **Then** 빈 상태 안내 메시지가 표시된다.
5. **Given** 존재하지 않는 날짜 형식으로 접근하면, **When** 화면이 로드되면, **Then** 잘못된 날짜 안내와 피드 목록으로 돌아가는 경로가 제공된다.

### User Story 3 - 운영자가 이슈의 현재 처리 상태를 파악한다 (Priority: P2)

운영자는 각 이슈가 `draft`, `approved`, `rejected` 중 어느 상태인지 한눈에 확인할 수 있어야 한다.

**Why this priority**: 상태 표시는 #8(승인·반려) 액션을 실행하기 전에 현재 진행 상황을 파악하기 위한 기반이다. #8이 없어도 단독으로 읽기 전용 상태 확인 가치가 있다.

**Independent Test**: 상태가 다른 여러 이슈가 있는 날짜에서 각 이슈마다 상태 배지가 정확하게 노출되면 통과.

**Acceptance Scenarios**:

1. **Given** 이슈가 `draft` 상태일 때, **When** 이슈 목록에서 해당 항목을 보면, **Then** `draft` 상태 배지가 표시된다.
2. **Given** 이슈가 `approved` 상태일 때, **When** 이슈 목록에서 해당 항목을 보면, **Then** `approved` 상태 배지가 표시된다.
3. **Given** 이슈가 `rejected` 상태일 때, **When** 이슈 목록에서 해당 항목을 보면, **Then** `rejected` 상태 배지가 표시된다.

---

### Edge Cases

- 피드 목록 API 호출 실패 시: 오류 안내 메시지와 재시도 동작을 제공한다.
- 날짜별 이슈 목록 API 호출 실패 시: 오류 안내 메시지와 피드 목록으로 돌아가는 경로를 제공한다.
- 특정 이슈의 `cards_data`가 비어 있거나 파싱 불가 상태일 때: 해당 이슈는 카드 미리보기 없이 "카드 데이터 없음" 안내로 목록에 표시한다.
- 잘못된 날짜 형식(`YYYY-MM-DD`가 아닌 경우) URL로 접근 시: 잘못된 날짜 안내를 노출하고 피드 목록으로 이동할 수 있는 경로를 제공한다.
- 피드 목록이 많아질 때: 최신 날짜 기준 정렬을 유지하며 페이지네이션 또는 스크롤로 성능을 유지한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 인증된 운영자에게 `/admin` 경로에서 날짜별 피드 목록을 제공해야 한다.
- **FR-002**: 피드 목록은 최신 날짜 우선으로 정렬되어야 한다.
- **FR-003**: 피드 목록의 각 항목은 날짜, 상태(`draft` / `published`), 이슈 수를 노출해야 한다.
- **FR-004**: `published` 상태 피드는 발행 시각을 추가로 노출해야 한다.
- **FR-005**: 피드 목록에서 항목을 선택하면 `/admin/feed/[date]` 화면으로 이동해야 한다.
- **FR-006**: 시스템은 인증된 운영자에게 `/admin/feed/[date]` 경로에서 해당 날짜의 이슈 목록을 제공해야 한다.
- **FR-007**: 이슈 목록의 각 항목은 이슈 제목, 엔티티 이름, 현재 상태(`draft` / `approved` / `rejected`), 카드 장 수를 노출해야 한다.
- **FR-008**: 운영자가 이슈 항목을 선택하면 해당 이슈의 카드 미리보기(카드 타입, 제목, 핵심 텍스트)가 노출되어야 한다.
- **FR-009**: 피드 목록과 이슈 목록 모두 데이터가 없는 경우 빈 상태 안내 메시지를 표시해야 한다.
- **FR-010**: 데이터 로드 중에는 로딩 상태를 표시해야 한다.
- **FR-011**: API 오류 발생 시 오류 안내와 복구 경로(재시도 또는 이전 화면 이동)를 제공해야 한다.
- **FR-012**: 잘못된 날짜 형식 URL 접근 시 안내 메시지와 피드 목록 이동 경로를 제공해야 한다.
- **FR-013**: 이슈 목록 화면은 해당 날짜 정보와 피드 목록으로 돌아가는 내비게이션을 포함해야 한다.

### Key Entities

- **Feed**: 특정 날짜의 이슈 모음. `date`, `status`(`draft` / `published`), `published_at`, 이슈 수를 가진다.
- **Issue**: 개별 이슈 카드 세트. `feed_id`, `entity_type`, `entity_name`, `title`, `status`(`draft` / `approved` / `rejected`), `cards_data`를 가진다.
- **Card**: Issue의 `cards_data` 내 개별 슬라이드. `type`, `title`, 타입별 텍스트 필드를 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 운영자가 `/admin`에 진입 후 3초 이내에 피드 목록이 표시된다.
- **SC-002**: 피드 목록에서 특정 날짜 항목을 선택한 뒤 3초 이내에 이슈 목록이 표시된다.
- **SC-003**: 피드 목록의 모든 항목에서 상태(`draft` / `published`) 표시 정확도가 100%다.
- **SC-004**: 이슈 목록의 모든 항목에서 상태 배지(`draft` / `approved` / `rejected`) 표시 정확도가 100%다.
- **SC-005**: 카드 데이터가 있는 이슈를 선택했을 때 카드 미리보기가 정상 노출되는 비율이 100%다.
- **SC-006**: 빈 상태, API 오류, 잘못된 날짜 형식 접근 시 100% 안내 메시지와 복구 경로가 표시된다.

## Assumptions

- 이 기능은 읽기 전용 화면이다. 이슈 편집, 승인·반려, 피드 발행은 각각 #8, #9에서 구현한다.
- `cards_data`는 기존에 정의된 JSON 스키마를 따른다고 가정하며, 미리보기는 스키마 파싱 가능 범위 내에서만 렌더링한다.
- Admin 화면은 운영자 1인 전용이며 데스크톱 화면 기준으로 설계한다. 모바일 최적화는 MVP 범위 밖이다.
- 피드 목록은 최근 30일 이내 항목을 기본으로 표시한다.
- 이슈 목록은 `draft` 이슈도 포함하여 운영자가 검토 대상 전체를 파악할 수 있도록 한다.

## Dependencies

- **선행**: #4 DB 스키마 마이그레이션 (완료) — `feeds`, `issues` 테이블 구조 기반
- **선행**: #5 카드 스키마 타입/검증 레이어 (완료) — `cards_data` 타입 안전성 기반
- **후행**: #8 Admin 이슈 편집/순서조정/승인·반려 — 이 화면의 이슈 목록에서 편집·승인 액션 연결
- **후행**: #9 Admin 피드 발행 워크플로우 — 이 화면의 피드 목록에서 발행 액션 연결
