## 배경 및 문제 정의

현재 파이프라인이 생성하는 카드뉴스는 7개 타입이 있지만, 모두 동일한 `제목 + 본문` 구조를 공유한다. AI가 타입과 무관하게 같은 패턴으로 글을 쓰게 되어, 독자는 두 번째 카드부터 내용을 예측하고 스와이프를 멈춘다.

**해결 목표**: 카드마다 완전히 다른 정보 구조를 강제해서, 각 카드가 서로 다른 방식으로 인사이트를 전달하도록 만든다.

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 다양한 형식으로 카드뉴스 소비 (Priority: P1)

독자(25-40세 직장인 투자자)가 오늘 생성된 카드뉴스를 읽을 때, 각 카드가 서로 다른 방식으로 정보를 전달한다. 첫 번째 카드는 "얼마나 변했는가(Δ)"를 숫자로 보여주고, 다음 카드는 "왜 그랬는가"를 결과→원인 순서로 설명하고, 그 다음은 충격적인 단일 통계, 비교 테이블 순으로 이어진다. 마지막 카드는 한 문장 결론으로 끝난다.

**Why this priority**: 카드 형식 다양성이 이 기능의 핵심 목표다. 이것이 구현되어야 나머지 개선 사항들이 의미를 갖는다.

**Independent Test**: 파이프라인을 한 번 실행해 생성된 카드들이 각기 다른 필드 구조를 갖는지 확인하면 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 파이프라인이 정상 실행되었을 때, **When** 생성된 카드뉴스를 열면, **Then** 첫 번째 카드에 "전 → 후" Δ 수치가 존재한다.
2. **Given** 카드 시퀀스를 볼 때, **When** 카드를 순서대로 넘기면, **Then** 연속된 두 카드가 동일한 정보 구조(같은 필드 조합)를 갖지 않는다.
3. **Given** 카드뉴스 한 편이 생성되었을 때, **When** 마지막 카드를 보면, **Then** 한 문장 결론과 3개 이하의 근거가 제시된다.
4. **Given** "커뮤니티 반응" 형식의 카드가 있을 때, **When** 내용을 확인하면, **Then** 출처 없는 가짜 커뮤니티 인용문이 없다.

---

### User Story 2 - 구체적 수치 기반의 신뢰할 수 있는 콘텐츠 (Priority: P2)

독자가 카드를 읽을 때, 모든 주장에 구체적인 수치 또는 비교 데이터가 함께 제시된다. "원화가 약세다"가 아니라 "유로 +10%, 파운드 +10%일 때 원화는 -X% — 17년 최저"처럼, 비교 맥락 속에서 수치를 제시한다.

**Why this priority**: 수치 기반 신뢰성은 카드 형식 다양성과 함께 독자 체류를 결정하는 핵심 요소다.

**Independent Test**: 생성된 카드에서 Δ 수치가 없는 `delta` 타입 카드를 찾으려 시도해 0건이면 통과.

**Acceptance Scenarios**:

1. **Given** 수치 변화를 나타내는 카드가 생성되었을 때, **When** 내용을 확인하면, **Then** "전 값", "후 값", "기간" 세 항목이 모두 채워져 있다.
2. **Given** 비교 카드가 생성되었을 때, **When** 내용을 확인하면, **Then** 2개 이상의 비교 항목이 방향(상승/하락)과 수치를 함께 보여준다.
3. **Given** 통계 카드가 생성되었을 때, **When** 내용을 확인하면, **Then** 수치 하나와 그 수치의 의미를 뒤집는 해석이 함께 제시된다.

---

### User Story 3 - 독자 생활과 연결되는 마지막 정보 (Priority: P2)

독자가 거시경제·종목 이슈를 읽고 나서, "그래서 내 지갑에는 얼마나 영향이 있지?"라는 질문에 답하는 카드를 보게 된다. 환율 이슈라면 해외직구·여행 비용 변화, 반도체 이슈라면 포트폴리오 영향이 구체적인 금액으로 나타난다.

**Why this priority**: 독자가 "이거 공유해야겠다"는 동기를 갖게 만드는 요소다. 단순 정보 전달을 넘어서는 차별화 포인트.

**Independent Test**: 환율 또는 소비 관련 이슈에서 독자 영향(impact) 카드가 1개 이상 포함되는지 확인.

**Acceptance Scenarios**:

1. **Given** 환율 또는 소비재 관련 이슈가 생성되었을 때, **When** 카드 목록을 확인하면, **Then** 독자 실생활 비용 변화가 담긴 카드가 포함되어 있다.
2. **Given** 독자 영향 카드가 있을 때, **When** 내용을 확인하면, **Then** 각 항목에 "변경 전 금액 → 변경 후 금액"이 구체적으로 제시된다.

---

### User Story 4 - 2단계 생성으로 팩트 정확도 향상 (Priority: P3)

운영자가 파이프라인 실행 로그를 확인할 때, 기사 원문 → 핵심 팩트 구조화 → 카드 생성의 두 단계가 명확히 구분되어 기록된다. 팩트 추출 단계가 실패해도 카드 생성이 원문으로 폴백(fallback)되거나 해당 이슈만 건너뛴다.

**Why this priority**: 직접 독자에게 보이지는 않지만, 카드 정확도와 운영 안정성을 높이는 기반 개선이다.

**Independent Test**: 팩트 추출 단계를 인위적으로 실패시켰을 때 나머지 이슈가 정상 생성되면 통과.

**Acceptance Scenarios**:

1. **Given** 파이프라인이 실행되었을 때, **When** 로그를 확인하면, **Then** 각 이슈에 대해 "팩트 추출 결과"가 별도 항목으로 기록된다.
2. **Given** 특정 이슈의 팩트 추출이 실패했을 때, **When** 파이프라인이 계속 실행되면, **Then** 다른 이슈 생성은 영향받지 않는다.

---

### Edge Cases

- 기사에 구체적인 수치가 없을 때: `delta` 타입 대신 `cause` 또는 `stat` 타입으로 대체 생성할 수 있는가?
- 이슈가 새로운 주체를 소개하는 경우(독자에게 생소한 회사/지표): `delta-intro` 타입으로 주체 설명이 포함되는가?
- 카드 수 제한(최대 7장) 내에서 `verdict`(결론)가 항상 마지막 카드인가?
- `compare` 타입에서 비교 행이 1개뿐인 경우: 최소 2개 이상을 강제하는가?
- 이미지 카테고리 키가 정의된 목록에 없는 것을 AI가 생성하는 경우: 기본값으로 폴백되는가?

## Requirements *(mandatory)*

### Functional Requirements

**카드 타입 시스템**

- **FR-001**: 시스템은 8개의 고유한 카드 타입을 지원해야 한다: `delta`, `delta-intro`, `cause`, `stat`, `compare`, `impact`, `verdict`, `question`
- **FR-002**: 각 카드 타입은 서로 다른 필드 구조를 가져야 하며, `title + body`를 공통 필드로 공유하지 않아야 한다
- **FR-003**: `delta` 타입 카드는 반드시 "전 값", "후 값", "기간", "한 줄 해석" 4개 필드를 모두 포함해야 한다
- **FR-004**: `cause` 타입 카드는 "결과"를 먼저, "원인"을 후에 배치하는 구조를 강제해야 한다
- **FR-005**: `stat` 타입 카드는 단일 수치, 수치 레이블, 상식을 뒤집는 해석 3개 필드를 포함해야 한다
- **FR-006**: `compare` 타입 카드는 질문, 비교 행 배열(최소 2개), 결론 문장을 포함해야 하며 각 행에 방향(상승/하락/최저) 정보가 있어야 한다
- **FR-007**: `impact` 타입 카드는 독자 실생활 항목별 "전 금액 → 후 금액 → 차이" 구조를 포함해야 한다
- **FR-008**: `verdict` 타입 카드는 한 문장 결론과 근거 목록(최대 3개)을 포함해야 하며, 이슈의 마지막 카드로 위치해야 한다
- **FR-009**: `question` 타입 카드는 독자 궁금증을 유발하는 질문과 힌트로 구성되며, 다음 카드로의 연결 고리 역할을 해야 한다
- **FR-010**: `delta-intro` 타입 카드는 delta 필드에 더해 "주체 이름", "주체 설명", "주목 이유" 필드를 포함해야 한다

**카드 생성 파이프라인**

- **FR-011**: 파이프라인은 카드 생성 이전에 기사 원문에서 핵심 팩트(수치 변화, 원인-결과, 주요 통계, 리스크)를 구조화하는 단계를 수행해야 한다
- **FR-012**: 팩트 구조화 단계는 비용이 낮은 AI 모델을 사용해야 하고, 카드 생성 단계는 품질이 높은 AI 모델을 사용해야 한다
- **FR-013**: 팩트 구조화 단계 실패 시 해당 이슈를 건너뛰고 나머지 이슈 생성을 계속해야 한다
- **FR-014**: 팩트 구조화 결과는 파이프라인 실행 로그에 별도 항목으로 기록되어야 한다

**이미지 카테고리 시스템**

- **FR-015**: 카드는 자유형 텍스트 검색어 대신 사전 정의된 이미지 카테고리 키를 참조해야 한다
- **FR-016**: AI가 정의되지 않은 이미지 카테고리 키를 생성할 경우 기본 카테고리로 폴백되어야 한다
- **FR-017**: 이미지 카테고리 키 목록은 최소 8개 카테고리를 포함해야 한다: `stock-up`, `stock-down`, `currency`, `ai-chip`, `earnings`, `warning`, `growth`, `consumer`

**콘텐츠 품질**

- **FR-018**: 생성된 카드의 이슈 유형(급등락/실적/거시/테마/교육)에 따라 카드 시퀀스가 다르게 구성되어야 한다
- **FR-019**: 커뮤니티 반응 카드 타입은 더 이상 생성되지 않아야 한다 (출처 없는 가짜 인용문 방지)
- **FR-020**: 각 카드의 텍스트 콘텐츠는 3줄 이내로 제한되어야 한다

### Key Entities

- **카드 타입(CardType)**: 8개 고유 타입 중 하나. 각 타입은 해당 타입에만 존재하는 고유 필드 구조를 가짐
- **팩트 구조(ExtractedFact)**: 기사 원문에서 추출된 핵심 정보. 수치 변화(Δ), 원인-결과, 핵심 통계, 리스크로 구성됨
- **이미지 카테고리(ImageCategory)**: 사전 정의된 카테고리 키. AI가 자유롭게 생성하지 않고 목록에서 선택
- **카드 시퀀스(CardSequence)**: 이슈 유형에 따른 카드 순서 규칙. 타입별 권장 순서가 다름

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 파이프라인이 생성한 카드뉴스에서 연속된 두 카드가 동일한 정보 구조를 갖는 비율이 10% 이하여야 한다
- **SC-002**: 생성된 이슈 중 첫 번째 카드(Δ 수치 카드)에 구체적 수치가 없는 비율이 5% 이하여야 한다
- **SC-003**: 팩트 구조화 단계 도입 후 카드당 AI 비용이 Phase 1 대비 동등하거나 낮아야 한다
- **SC-004**: 이미지 카테고리 키 오류(정의되지 않은 키 생성) 비율이 2% 이하여야 한다
- **SC-005**: 생성된 카드 중 `community` 타입 카드가 0건이어야 한다
- **SC-006**: 팩트 구조화 단계 실패 시 전체 파이프라인 실패 없이 해당 이슈만 스킵되어야 한다 (파이프라인 성공률 100% 유지)
- **SC-007**: 생성된 카드뉴스 한 편에서 `verdict` 타입 카드가 마지막 위치에 있는 비율이 95% 이상이어야 한다

## Assumptions

- Phase 1(#98) 머지가 완료된 상태에서 진행한다
- 기존 DB에 저장된 카드 데이터는 마이그레이션하지 않는다 (신규 생성분부터 새 타입 적용)
- 이미지 카테고리 풀의 실제 이미지 URL은 구현 단계에서 큐레이션한다 (초기에는 Unsplash URL 활용 가능)
- 렌더링 컴포넌트(프론트엔드)는 이 기능의 범위에 포함된다 — 새 타입을 화면에 표시할 수 있어야 한다
- `source` 타입 카드는 현재 방식을 유지한다 (변경 없음)
