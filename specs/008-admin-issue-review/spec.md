## User Scenarios & Testing *(mandatory)*

### User Story 1 - 이슈 승인 및 반려 (Priority: P1)

운영자는 AI가 생성한 이슈 초안을 검토한 후 개별 이슈 단위로 승인하거나 반려한다.
승인된 이슈만 피드 발행 대상이 되며, 반려된 이슈는 발행 대상에서 제외된다.

**Why this priority**: 승인/반려는 운영 검수 워크플로우의 핵심이다. 편집 없이 승인/반려만 동작해도 피드 발행 흐름 전체가 성립한다.

**Independent Test**: 특정 날짜 피드 검토 화면에서 이슈 승인 버튼을 누르면 해당 이슈의 상태가 "승인됨"으로 변경되고, 반려 버튼을 누르면 "반려됨"으로 변경됨을 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** draft 상태의 이슈가 검토 화면에 표시되어 있을 때, **When** 운영자가 승인 버튼을 누르면, **Then** 해당 이슈의 상태가 즉시 "승인됨(approved)"으로 전환되고 UI에 반영된다.
2. **Given** draft 상태의 이슈가 검토 화면에 표시되어 있을 때, **When** 운영자가 반려 버튼을 누르면, **Then** 해당 이슈의 상태가 즉시 "반려됨(rejected)"으로 전환되고 UI에 반영된다.
3. **Given** approved 상태인 이슈가 있을 때, **When** 운영자가 반려 버튼을 누르면, **Then** 상태가 rejected로 재전환된다 (상태 재변경 가능).
4. **Given** 인증되지 않은 사용자가 있을 때, **When** 이슈 승인/반려 요청을 보내면, **Then** 요청이 거부되고 로그인 화면으로 이동한다.
5. **Given** 승인/반려 처리가 진행 중일 때, **When** 동일 버튼을 다시 누르면, **Then** 처리 완료 전까지 버튼이 비활성화되어 중복 요청이 차단된다.

### User Story 2 - 카드 텍스트 수정 (Priority: P2)

운영자는 AI가 생성한 카드 내의 텍스트(제목, 본문, 태그 등)를 직접 수정하여 품질을 보완할 수 있다.

**Why this priority**: 텍스트 편집은 AI 생성 품질 보완에 필요하지만, 편집 없이 승인/반려만 해도 워크플로우가 동작한다.

**Independent Test**: 이슈 상세에서 특정 카드의 제목을 수정 후 저장하면, 해당 이슈의 카드 데이터가 갱신되어 다음 열람 시 수정된 텍스트가 표시됨을 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 이슈 카드 목록이 펼쳐진 상태일 때, **When** 운영자가 카드의 텍스트 필드(제목, 본문, 태그)를 수정하고 저장하면, **Then** 변경 사항이 저장되고 화면에 즉시 반영된다.
2. **Given** 필수 텍스트 필드를 빈 값으로 지운 채 저장을 시도할 때, **When** 저장 버튼을 누르면, **Then** 저장이 거부되고 어떤 필드가 비어 있는지 안내 메시지가 표시된다.
3. **Given** 텍스트를 수정 중일 때, **When** 취소 버튼을 누르면, **Then** 마지막 저장 상태로 복원된다.
4. **Given** 저장 중 오류가 발생하면, **Then** 사용자에게 실패 알림이 표시되고 수정 내용이 유실되지 않는다.

### User Story 3 - 카드 순서 조정 (Priority: P3)

운영자는 이슈 내 카드의 순서를 위/아래로 이동하여 최종 노출 순서를 조정할 수 있다.

**Why this priority**: 카드 순서는 AI가 결정하지만 운영자가 미세 조정할 수 있어야 한다. 승인/반려, 텍스트 편집 이후의 보완 기능이다.

**Independent Test**: 이슈 카드 목록에서 두 번째 카드를 위로 이동하면 첫 번째 자리로 이동하고, 저장 후 다시 열람해도 동일한 순서가 유지됨을 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 이슈에 카드가 2장 이상 있을 때, **When** 운영자가 카드의 이동 버튼(위/아래)을 누르면, **Then** 해당 카드가 인접 카드와 순서가 바뀌고 즉시 UI에 반영된다.
2. **Given** 첫 번째 카드가 있을 때, **When** 해당 카드의 '위로 이동' 버튼을 확인하면, **Then** 버튼이 비활성화되어 있어 클릭할 수 없다.
3. **Given** 마지막 카드가 있을 때, **When** 해당 카드의 '아래로 이동' 버튼을 확인하면, **Then** 버튼이 비활성화되어 있어 클릭할 수 없다.
4. **Given** 순서 변경 후 저장 버튼을 누르면, **Then** 변경된 순서가 저장되어 다음 열람 시 동일한 순서로 표시된다.

---

### Edge Cases

- 동시 편집: 운영자가 이슈를 편집 중일 때 다른 세션에서 동일 이슈를 수정하면 마지막 저장이 최종값이 된다 (MVP: 동시 편집 잠금 없음).
- 카드가 1장인 이슈에서는 위/아래 이동 버튼이 모두 비활성화되어야 한다.
- 카드 데이터가 손상되어 파싱되지 않는 이슈에서 편집 시도 시 명확한 오류 메시지를 표시하고 편집 UI를 비활성화한다.
- 네트워크 오류로 저장 실패 시 수정 내용이 유실되지 않도록 실패 알림과 재시도 안내를 제공한다.
- 승인/반려 버튼 연속 클릭 시 처리 중 버튼 비활성화로 중복 요청을 방지한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 운영자는 이슈별로 "승인(approved)" 또는 "반려(rejected)" 상태 변경을 수행할 수 있어야 한다.
- **FR-002**: 이슈 상태 변경(승인/반려)은 즉시 저장되며 화면에 실시간으로 반영되어야 한다.
- **FR-003**: 이슈의 상태는 draft ↔ approved ↔ rejected 간 재변경이 가능해야 한다.
- **FR-004**: 상태 변경 처리 중에는 해당 버튼이 비활성화되어 중복 요청이 방지되어야 한다.
- **FR-005**: 운영자는 이슈 내 개별 카드의 텍스트 필드(제목, 본문, 태그)를 수정할 수 있어야 한다. 태그는 단일 텍스트 입력란에 쉼표로 구분하여 입력한다 (예: `주식,투자,반도체`).
- **FR-006**: 텍스트 수정 시 필수 필드가 비어 있는 경우 저장이 거부되고 안내 메시지가 표시되어야 한다. 태그는 MVP에서 필수 필드로 간주하지 않는다.
- **FR-007**: 텍스트 수정 취소 시 마지막 저장 상태로 복원되어야 한다.
- **FR-008**: 운영자는 이슈 내 카드의 순서를 위/아래로 1칸씩 이동할 수 있어야 한다.
- **FR-009**: 첫 번째 카드의 '위로 이동' 버튼과 마지막 카드의 '아래로 이동' 버튼은 비활성화되어야 한다.
- **FR-010**: 카드 텍스트 수정 및 순서 변경은 각 카드별 독립적인 저장 버튼으로 확정되어야 한다. 카드 단위로 저장이 이루어지며, 저장 실패 시 해당 카드에만 영향이 한정된다.
- **FR-011**: 모든 편집/상태변경 기능은 인증된 운영자만 접근 가능해야 한다.
- **FR-012**: 카드 데이터가 손상된 이슈에서 편집 시도 시 명확한 오류 메시지를 표시하고 편집을 차단해야 한다.
- **FR-013**: 저장 중 오류 발생 시 사용자에게 실패 알림과 재시도 수단을 제공해야 한다.

### Key Entities

- **Issue(이슈)**: AI가 생성한 뉴스 카드 묶음. 상태(draft/approved/rejected), 피드 내 노출 순서, 카드 배열을 가진다.
- **Card(카드)**: 이슈를 구성하는 개별 슬라이드. 타입(cover/reason/bullish/bearish/community/stats/source)별로 제목, 본문, 태그 등의 텍스트 필드와 시각 정보를 가지며, 이슈 내에서 순서가 있다.
- **Feed(피드)**: 특정 날짜의 이슈 묶음. 이슈들은 하나의 피드에 속한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 운영자가 승인 또는 반려 버튼을 누른 후 2초 이내에 상태 변경이 화면에 반영된다.
- **SC-002**: 운영자가 카드 텍스트 수정 후 저장하면 2초 이내에 저장 완료 확인이 표시된다.
- **SC-003**: 운영자가 이슈 1건(카드 5장 기준)에 대한 검토 전체(텍스트 수정 + 순서 조정 + 승인)를 5분 이내에 완료할 수 있다.
- **SC-004**: 저장 실패 시 100%의 경우에 사용자에게 실패 알림이 표시된다.
- **SC-005**: 비인증 접근 시 100% 거부된다.
- **SC-006**: 승인/반려 버튼 중복 클릭으로 인한 중복 상태 변경이 0건 발생한다.

## Clarifications

### Session 2026-03-04

- Q: 저장 액션의 범위는 카드별인가 이슈별인가? → A: 카드별 저장. 각 카드마다 독립적인 저장 버튼을 가지며, 텍스트 수정과 순서 변경 모두 카드 단위로 확정된다.
- Q: 태그 필드의 입력 형식은? → A: 쉼표 구분 자유 텍스트. 단일 입력란에 쉼표로 구분 입력. MVP에서 태그는 필수 필드가 아니다.
- Q: 승인/반려 감사 로그가 필요한가? → A: MVP 제외. DB `updated_at` 타임스탬프로 대체. 별도 이력 테이블은 Out of Scope.

## Assumptions

- 운영자는 1인이며, 동시 다중 세션 편집은 MVP에서 고려하지 않는다. 마지막 저장이 최종 값이다.
- 카드 스키마(타입, 필수 필드 구조)는 고정이며, 운영자가 카드 타입 자체를 변경하거나 카드를 추가/삭제하는 것은 MVP 범위 외다.
- 카드의 `visual` 필드(색상 코드)는 MVP에서 편집 대상에서 제외한다.
- 이슈 간 노출 순서(피드 내 이슈 순서) 조정은 #9 피드 발행 워크플로우 범위다. 이 기능에서는 이슈 내 카드 순서만 조정한다.
- 카드 텍스트 수정 UI는 이슈 펼치기(accordion) 내부에서 인라인으로 제공된다.

## Dependencies

- **선행**: #7 Admin 피드 목록/날짜별 이슈 검토 화면 (완료) — 이슈 목록 조회 및 펼치기 기능 제공
- **후행**: #9 Admin 피드 발행 워크플로우 — 승인된 이슈를 묶어 발행하는 기능이 이 기능에 의존한다.

## Out of Scope

- 이슈 간 display_order 조정 (피드 내 이슈 순서) → #9
- 카드 추가/삭제
- 카드 타입 변경
- 카드 visual(색상) 편집
- 동시 편집 충돌 해결 (낙관적 락 등)
- 이슈 재생성 또는 삭제
- 승인/반려 감사 로그 (별도 이력 테이블) — DB `updated_at` 타임스탬프로 대체
