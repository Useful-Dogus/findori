## User Scenarios & Testing *(mandatory)*

### User Story 1 - 운영자가 보호된 Admin 화면에 안전하게 진입한다 (Priority: P1)

운영자는 비밀번호를 입력해 인증한 뒤 `/admin` 이하 화면에 접근할 수 있어야 한다. 인증되지 않은 상태에서는 보호된 운영 화면이 노출되면 안 된다.

**Why this priority**: Admin 영역 보호가 없으면 이후 피드 검토, 발행, 매체 관리 기능이 모두 무방비 상태가 된다. 이 기능은 `#7~#10`의 선행 조건이다.

**Independent Test**: 인증되지 않은 브라우저에서 `/admin` 접근 시 로그인 화면으로 이동하고, 올바른 비밀번호 입력 후 다시 `/admin`에 접근 가능해지면 통과.

**Acceptance Scenarios**:

1. **Given** 운영자가 인증되지 않은 상태일 때, **When** `/admin` 또는 그 하위 경로에 접근하면, **Then** 로그인 화면으로 리다이렉트된다.
2. **Given** 운영자가 로그인 화면에 있을 때, **When** 올바른 비밀번호를 제출하면, **Then** 보호된 세션이 발급되고 `/admin` 진입이 허용된다.
3. **Given** 운영자가 이미 유효한 세션을 보유하고 있을 때, **When** `/admin/feed/2026-02-26` 같은 보호된 경로에 접근하면, **Then** 로그인 화면을 거치지 않고 해당 화면으로 진입한다.

---

### User Story 2 - 운영자가 세션 만료 및 로그아웃을 예측 가능하게 처리한다 (Priority: P2)

운영자는 로그인 후 일정 기간 동안 다시 로그인하지 않고 작업을 이어갈 수 있어야 하며, 직접 로그아웃하면 즉시 보호 영역 접근 권한이 제거되어야 한다.

**Why this priority**: 운영자 1인용 MVP라도 세션 수명과 로그아웃 동작이 불명확하면 운영 흐름이 흔들리고, 공유 기기나 장시간 방치 상황에서 보안 리스크가 남는다.

**Independent Test**: 로그인 후 브라우저를 새로고침하거나 새 탭을 열어도 보호 경로 접근이 유지되고, 로그아웃 직후 같은 경로 접근 시 다시 로그인 화면으로 이동하면 통과.

**Acceptance Scenarios**:

1. **Given** 운영자가 정상 로그인한 상태일 때, **When** 브라우저 새로고침 또는 새 탭으로 `/admin`에 재접근하면, **Then** 유효 기간 내에는 세션이 유지된다.
2. **Given** 운영자가 로그인한 상태일 때, **When** 로그아웃을 실행하면, **Then** 세션이 즉시 제거되고 이후 `/admin` 접근 시 로그인 화면으로 이동한다.
3. **Given** 운영자의 세션이 만료되었을 때, **When** 보호된 경로에 접근하면, **Then** 접근이 거부되고 로그인 화면으로 이동한다.

---

### User Story 3 - 운영자가 실패 상황을 이해하고 다시 시도할 수 있다 (Priority: P3)

운영자는 잘못된 비밀번호 입력, 손상된 세션, 비정상 접근 같은 실패 상황에서도 민감한 정보 노출 없이 명확한 실패 결과와 재진입 경로를 받아야 한다.

**Why this priority**: 운영 도구는 실패 시에도 상태가 분명해야 한다. 특히 인증 실패와 세션 이상 상태를 구분 없이 처리하면 디버깅과 운영이 어려워진다.

**Independent Test**: 잘못된 비밀번호 제출, 변조된 세션 쿠키, 인증 없이 Admin API 호출을 각각 시도해 일관된 거부 응답과 재시도 경로가 제공되면 통과.

**Acceptance Scenarios**:

1. **Given** 운영자가 로그인 화면에 있을 때, **When** 잘못된 비밀번호를 제출하면, **Then** 로그인은 실패하고 보호된 세션은 발급되지 않으며 재시도 가능한 실패 메시지가 노출된다.
2. **Given** 유효하지 않거나 손상된 세션 정보가 전달될 때, **When** 보호된 화면 또는 API에 접근하면, **Then** 요청은 거부되고 세션은 무효 처리된다.
3. **Given** 인증되지 않은 요청이 Admin API를 호출할 때, **When** 서버가 요청을 검증하면, **Then** 데이터는 반환되지 않고 인증 필요 응답이 반환된다.

### Edge Cases

- 운영자가 이미 로그인한 상태에서 다시 로그인 화면에 접근하는 경우: 불필요한 재인증 없이 Admin 기본 화면으로 보낸다.
- 로그인 요청 본문이 비어 있거나 비밀번호 필드 형식이 잘못된 경우: 인증 실패와 구분되는 잘못된 요청으로 처리한다.
- 세션 쿠키는 존재하지만 서명 검증 또는 만료 검증에 실패하는 경우: 세션을 신뢰하지 않고 즉시 제거 후 로그인 화면으로 이동한다.
- 로그아웃 요청 시 이미 세션이 없는 경우: 오류 없이 로그아웃 완료 상태로 처리한다.
- 공개 화면과 공개 API는 Admin 인증과 무관하게 계속 접근 가능해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 `/admin` 이하 보호 경로에 대해 인증되지 않은 사용자의 접근을 차단해야 한다.
- **FR-002**: 시스템은 인증되지 않은 사용자가 보호된 Admin 화면에 접근하면 `/admin/login`으로 이동시켜야 한다.
- **FR-003**: 시스템은 운영자가 비밀번호를 제출해 인증을 시도할 수 있는 로그인 진입점을 제공해야 한다.
- **FR-004**: 시스템은 제출된 비밀번호가 운영용 비밀번호와 일치할 때만 로그인 성공으로 처리해야 한다.
- **FR-005**: 시스템은 로그인 성공 시 클라이언트 스크립트에서 읽을 수 없는 보호된 세션을 발급해야 한다.
- **FR-006**: 시스템은 발급된 세션에 대해 다음 보안 속성을 보장해야 한다: 브라우저 스크립트 비노출, HTTPS 환경에서만 전송, 교차 사이트 요청 제한.
- **FR-007**: 시스템은 발급된 세션의 유효 기간을 7일로 유지해야 하며, 유효 기간 경과 후에는 재인증을 요구해야 한다.
- **FR-008**: 시스템은 유효한 세션이 있는 운영자에 대해 `/admin`, `/admin/feed/[date]`, `/admin/sources` 및 이후 추가되는 Admin 하위 경로 접근을 허용해야 한다.
- **FR-009**: 시스템은 인증되지 않았거나 세션이 유효하지 않은 요청이 Admin API에 접근하면 데이터를 반환하지 않고 인증 실패 응답을 반환해야 한다.
- **FR-010**: 시스템은 로그인 실패 시 비밀번호가 틀렸다는 사실 외 추가 민감 정보를 노출하지 않아야 하며, 운영자가 즉시 다시 시도할 수 있게 해야 한다.
- **FR-011**: 시스템은 운영자가 명시적으로 로그아웃할 수 있는 동작을 제공해야 하며, 로그아웃 후 기존 세션을 즉시 무효화해야 한다.
- **FR-012**: 시스템은 인증과 세션 처리에 필요한 운영 비밀값을 코드에 하드코딩하지 않고 런타임 구성으로만 읽어야 한다.
- **FR-013**: 시스템은 공개 피드 화면과 공개 API 요청에는 Admin 인증을 요구하지 않아야 한다.
- **FR-014**: 시스템은 손상되었거나 만료된 세션을 감지하면 해당 세션을 더 이상 신뢰하지 않고 로그인 흐름으로 복귀시켜야 한다.

### Key Entities

- **Admin Credential**: 운영자 인증에 사용하는 단일 비밀번호 규칙과 비교 대상 값.
- **Admin Session**: 운영자 로그인 성공 후 발급되는 보호된 인증 상태. 발급 시각, 만료 시각, 검증 가능 상태를 가진다.
- **Protected Admin Route**: 운영자 인증이 필요한 화면 경로 집합. MVP에서는 `/admin` 및 그 하위 경로가 대상이다.
- **Protected Admin Request**: 운영자 인증이 필요한 API 요청 집합. 인증 실패 시 데이터 접근이 차단된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 인증되지 않은 상태에서 보호된 Admin 화면 3종(`/admin`, 날짜별 검토 화면, 매체 관리 화면`)에 접근할 때 100% 로그인 화면으로 이동한다.
- **SC-002**: 올바른 비밀번호로 로그인한 운영자는 1분 이내에 Admin 기본 화면에 진입할 수 있다.
- **SC-003**: 로그인 성공 후 브라우저 새로고침 및 새 탭 재진입 시 세션 유효 기간 내에는 100% 재로그인 없이 접근이 유지된다.
- **SC-004**: 잘못된 비밀번호, 만료 세션, 손상된 세션으로 수행한 인증 시도는 100% 보호된 화면 및 데이터 접근을 차단한다.
- **SC-005**: 로그아웃 직후 같은 브라우저에서 보호된 Admin 경로에 다시 접근하면 100% 로그인 화면으로 복귀한다.
- **SC-006**: 공개 피드 경로와 공개 API 경로는 Admin 인증 기능 도입 후에도 기존과 동일하게 비로그인 접근이 가능하다.

## Assumptions

- MVP의 운영자는 1인이며, 계정 목록 관리나 다중 권한 체계는 포함하지 않는다.
- 로그인 방식은 환경변수 기반 단일 비밀번호 인증으로 고정하며, SSO나 외부 인증 제공자는 범위 밖이다.
- 로그인 실패 횟수 제한, CAPTCHA, 2단계 인증은 MVP 범위에 포함하지 않는다.
- 로그인 성공 후 이동 대상은 Admin 기본 화면이며, 보호 경로에 직접 진입한 경우에는 원래 요청한 경로 복귀를 허용한다.
- 공개 사용자용 화면과 공유 링크 열람 흐름은 이 기능의 영향 범위 밖이며, 보호 대상은 Admin 영역으로 한정한다.
