# Feature Specification: Cron 파이프라인 엔드포인트

**Feature Branch**: `011-cron-pipeline`
**Created**: 2026-03-03
**Status**: Draft
**Input**: User description: "#11 Cron 파이프라인 엔드포인트 구현"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 일일 자동 콘텐츠 수집·생성 (Priority: P1)

매일 오후 10시, 예약 스케줄러가 파이프라인 엔드포인트를 호출한다. 엔드포인트는 등록된 모든 활성 매체에서 당일 신규 기사를 수집하고, AI를 통해 이슈 카드 초안을 생성한 뒤 데이터베이스에 저장한다. 이를 통해 운영자는 다음 날 아침 Admin UI에서 바로 카드 초안을 검토·승인할 수 있다.

**Why this priority**: 파이프라인의 핵심 자동화 기능이다. 수동 개입 없이 매일 콘텐츠를 생산하는 것이 서비스 운영의 전제 조건이다.

**Independent Test**: 예약 호출 시뮬레이션으로 테스트 가능. 유효한 인증 토큰과 함께 엔드포인트를 호출하면, 기사 수집 → 카드 생성 → DB 저장이 완료되고 응답에 실행 요약이 포함된다.

**Acceptance Scenarios**:

1. **Given** 활성 매체가 1개 이상 등록되어 있고 유효한 인증 토큰으로 요청이 들어올 때, **When** 엔드포인트가 호출되면, **Then** 당일 신규 기사를 수집하고 이슈 카드 초안을 생성하여 `draft` 상태로 저장하고, 실행 결과 요약(처리 건수, 오류 수)을 응답으로 반환한다.
2. **Given** 파이프라인이 성공적으로 완료되었을 때, **When** 운영자가 Admin UI를 열면, **Then** 당일 생성된 이슈 카드 초안 목록이 표시된다.
3. **Given** 동일 날짜에 파이프라인이 이미 실행되었을 때, **When** 엔드포인트가 다시 호출되면, **Then** 중복 기사를 건너뛰고 새로 수집된 기사만 처리한다.

---

### User Story 2 - 인증되지 않은 호출 차단 (Priority: P1)

엔드포인트는 비밀 토큰 없이 들어오는 모든 요청을 거부한다. 운영자는 허가된 스케줄러만 파이프라인을 실행할 수 있다는 것을 확신할 수 있다.

**Why this priority**: 파이프라인은 외부 API 비용을 발생시키는 작업이므로 무단 호출 방지가 보안·비용 측면에서 필수다.

**Independent Test**: 토큰 없이 또는 잘못된 토큰으로 엔드포인트를 호출하면 인증 오류 응답이 반환되고 파이프라인이 실행되지 않는다.

**Acceptance Scenarios**:

1. **Given** 인증 토큰이 없는 요청이 들어올 때, **When** 엔드포인트가 호출되면, **Then** 인증 오류(401)를 반환하고 파이프라인을 실행하지 않는다.
2. **Given** 잘못된 인증 토큰이 포함된 요청이 들어올 때, **When** 엔드포인트가 호출되면, **Then** 인증 오류를 반환하고 파이프라인을 실행하지 않는다.
3. **Given** 유효한 인증 토큰이 포함된 요청이 들어올 때, **When** 엔드포인트가 호출되면, **Then** 정상적으로 파이프라인을 실행한다.

---

### User Story 3 - 부분 실패 허용 처리 (Priority: P2)

파이프라인 실행 중 일부 매체에서 기사 수집이 실패하거나 AI 카드 생성이 실패해도, 성공한 항목은 그대로 저장된다. 운영자는 부분 성공 결과를 검토할 수 있고, 실패 원인을 실행 로그에서 확인할 수 있다.

**Why this priority**: 특정 매체 장애가 전체 파이프라인을 중단시켜서는 안 된다. 부분 결과라도 콘텐츠가 생산되어야 운영이 유지된다.

**Independent Test**: 일부 매체를 응답 불가 상태로 만든 후 파이프라인을 실행하면, 정상 매체에서 수집된 카드는 저장되고 실패 정보가 로그에 기록된다.

**Acceptance Scenarios**:

1. **Given** 등록된 매체 중 일부가 응답 불가일 때, **When** 파이프라인이 실행되면, **Then** 실패 매체를 건너뛰고 정상 매체에서 수집된 기사로 카드를 생성·저장한다.
2. **Given** 일부 기사의 AI 카드 생성이 실패할 때, **When** 파이프라인이 실행되면, **Then** 해당 기사는 생략하고 성공한 나머지 기사의 카드를 저장하며, 실패 항목을 로그에 기록한다.
3. **Given** 파이프라인이 부분 실패로 완료되었을 때, **When** 응답이 반환되면, **Then** 성공 건수와 실패 건수가 모두 포함된 실행 요약을 반환한다.

---

### User Story 4 - 파이프라인 실행 로그 조회 (Priority: P3)

운영자는 과거 파이프라인 실행 이력을 조회하여 각 실행의 성공 여부, 처리 건수, 오류 내용을 확인할 수 있다.

**Why this priority**: 자동화 운영에서 실행 이력 가시성은 문제 진단과 신뢰성 확인에 필요하지만, 핵심 기능은 아니다.

**Independent Test**: 파이프라인이 실행된 후 로그 조회 API를 호출하면 해당 실행 기록이 목록에 포함된다.

**Acceptance Scenarios**:

1. **Given** 파이프라인이 1회 이상 실행된 후, **When** 운영자가 로그 목록을 조회하면, **Then** 각 실행의 시작 시간, 완료 시간, 상태(성공/실패/부분 성공), 처리 건수가 포함된 목록이 반환된다.
2. **Given** 파이프라인 실행 중 오류가 발생했을 때, **When** 운영자가 해당 실행 로그를 조회하면, **Then** 어떤 매체 또는 항목에서 어떤 오류가 발생했는지 상세 정보가 포함되어 있다.

---

### Edge Cases

- 당일 수집된 신규 기사가 하나도 없을 때 → 파이프라인은 정상 완료로 처리하고, 빈 실행 결과를 로그에 기록한다.
- 모든 매체가 응답 실패일 때 → 파이프라인 전체 실패로 처리하고, 로그에 실패 상태를 기록한다.
- 기사 본문이 AI 처리 한계를 초과할 때 → 해당 기사를 건너뛰고 로그에 기록한다.
- 파이프라인이 이미 실행 중인 상태에서 중복 호출이 들어올 때 → 중복 실행 방지를 위해 새 실행을 시작하지 않고 이미 실행 중임을 알리는 응답을 반환한다.
- DB 저장에 실패할 때 → 해당 항목은 손실되고 로그에 기록된다. 파이프라인은 다음 항목을 계속 처리한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 엔드포인트는 요청에 포함된 비밀 토큰을 검증한 후에만 파이프라인을 실행해야 한다.
- **FR-002**: 유효하지 않은 토큰이 포함된 요청은 즉시 거부(인증 오류 응답)해야 한다.
- **FR-003**: 파이프라인 실행 시 데이터베이스에 등록된 모든 활성 매체에서 당일 신규 기사를 수집해야 한다.
- **FR-004**: 동일 URL의 기사는 중복으로 처리하지 않아야 한다.
- **FR-005**: 수집된 기사를 AI 서비스를 통해 사전 정의된 카드 스키마에 맞는 이슈 카드 데이터로 변환해야 한다.
- **FR-006**: 생성된 이슈 카드는 `draft` 상태로 데이터베이스에 저장해야 한다.
- **FR-007**: 개별 매체 수집 실패가 전체 파이프라인을 중단시켜서는 안 된다.
- **FR-008**: 개별 기사의 AI 카드 생성 실패가 전체 파이프라인을 중단시켜서는 안 된다.
- **FR-009**: 각 파이프라인 실행의 시작 시간, 완료 시간, 상태, 처리 건수, 오류 정보를 로그로 기록해야 한다.
- **FR-010**: 파이프라인 완료 시 실행 결과 요약(성공 건수, 실패 건수, 소요 시간)을 응답으로 반환해야 한다.
- **FR-011**: 파이프라인이 이미 실행 중인 경우 중복 실행을 방지해야 한다.

### Key Entities

- **파이프라인 실행 로그**: 각 파이프라인 실행의 기록. 실행 ID, 시작 시간, 완료 시간, 상태(진행 중/성공/실패/부분 성공), 처리된 기사 수, 생성된 카드 수, 오류 목록을 포함한다.
- **뉴스 매체(Source)**: 기사를 수집하는 출처. 매체 URL, 활성 여부, 마지막 수집 성공 시간을 포함한다.
- **수집 기사(Article)**: 매체에서 가져온 원본 기사. URL(중복 판별 기준), 제목, 본문, 수집 날짜를 포함한다.
- **이슈 카드(Issue)**: AI가 생성한 구조화된 카드 데이터 묶음. 상태(`draft` / `approved` / `rejected` / `published`), 카드 데이터 배열, 연관 매체 정보를 포함한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 정기 예약 실행의 99% 이상이 수동 개입 없이 완료된다.
- **SC-002**: 파이프라인 호출 후 30분 이내에 이슈 카드 초안이 Admin UI에서 검토 가능한 상태가 된다.
- **SC-003**: 인증되지 않은 호출은 100% 차단된다.
- **SC-004**: 일부 매체 장애가 발생해도 정상 매체에서 수집된 콘텐츠가 유실되지 않는다 (부분 성공률 100%).
- **SC-005**: 파이프라인 실행 오류는 실행 완료 후 5분 이내에 로그에서 확인 가능하다.

## Assumptions *(optional)*

- 파이프라인은 일 1회 정기 실행이 기본이며, 추가적으로 Admin이 수동 트리거를 통해 재실행할 수 있다.
- 인증 방식은 `Authorization: Bearer <TOKEN>` 헤더를 사용하며, 토큰은 환경변수로 관리된다.
- 기사 중복 판별 기준은 URL이다.
- 수집 대상은 데이터베이스에 활성 상태로 등록된 매체로 한정된다.
- 카드 스키마는 사전에 정의되어 있으며, 파이프라인은 이 스키마 구조를 그대로 따른다.
- 파이프라인 로그는 데이터베이스에 영구 보관된다.
