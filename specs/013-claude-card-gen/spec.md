## User Scenarios & Testing *(mandatory)*

### User Story 1 - 운영자가 Admin에서 당일 이슈 카드 초안을 확인한다 (Priority: P1)

파이프라인이 실행된 뒤, 운영자가 Admin UI에 접속하면 당일 수집된 기사로 생성된 이슈 카드 초안 목록이 나타난다. 각 이슈는 제목·종목·카드 수를 표시하며, 카드를 열어보면 커버·원인·상승논거·하락논거·커뮤니티·출처 순서의 스토리가 구성되어 있다. 운영자는 내용을 검토하고 수정·승인·반려 결정을 내릴 수 있다.

**Why this priority**: 파이프라인의 최종 산출물. 이것이 없으면 Admin 검토 및 발행 워크플로우 전체가 무의미하다.

**Independent Test**: 수집된 기사 데이터가 존재할 때 생성 모듈을 실행하면, 각 이슈에 대해 스키마를 준수하는 cards[] 초안이 DB에 저장되고 Admin 목록에 나타난다.

**Acceptance Scenarios**:

1. **Given** 당일 수집된 기사가 1개 이상 존재할 때, **When** 생성 모듈이 실행되면, **Then** 기사 내용을 반영한 이슈 초안이 1개 이상 생성되어 `draft` 상태로 저장된다.
2. **Given** 생성된 이슈 초안의 카드 목록을 확인할 때, **When** 카드 순서와 타입을 검사하면, **Then** 첫 번째 카드는 커버(cover) 타입이고 마지막 카드는 출처(source) 타입이며 전체 카드 수는 3~7장이다.
3. **Given** 기사에 수치·근거가 포함된 경우, **When** 해당 기사가 reason/bullish/bearish 카드로 변환되면, **Then** 그 카드에는 최소 1개의 출처 항목이 포함된다.
4. **Given** 생성된 카드의 시각 디자인 필드를 확인할 때, **When** visual 필드의 색상값을 검사하면, **Then** 모든 색상값은 유효한 hex 코드 형식이며 Tailwind 클래스 문자열이 포함되지 않는다.

### User Story 2 - 시장 맥락 카드(코스피·나스닥·USD/KRW)가 매일 생성된다 (Priority: P2)

운영자가 당일 피드를 확인하면, 종목/테마 이슈와 별개로 코스피·나스닥·달러-원 환율 등 핵심 시장 지수 맥락 카드가 포함되어 있다. 이 카드는 개별 종목 이슈와 동일한 스키마를 따르지만 지수/환율 전용 분류로 구분된다.

**Why this priority**: SRS에서 "별도 호출로 생성"으로 명시된 독립 요구사항. 일반 이슈 생성과 입력 데이터 성격이 달라 독립적으로 처리된다.

**Independent Test**: 코스피·나스닥·USD/KRW 당일 수치 데이터를 입력으로 받아 생성 모듈을 실행하면, 지수 또는 환율 분류의 이슈 초안이 각각 생성된다.

**Acceptance Scenarios**:

1. **Given** 당일 시장 지수 데이터(코스피, 나스닥, USD/KRW)가 제공될 때, **When** 맥락 카드 생성이 실행되면, **Then** 각각 지수 또는 환율 분류의 이슈 초안이 생성된다.
2. **Given** 맥락 카드 이슈를 확인할 때, **When** 카드 구성을 검토하면, **Then** 종목 이슈와 동일한 스키마 제약(3~7장, cover 시작, source 종료)을 따른다.

---

### User Story 3 - 카드 생성 실패 시 파이프라인이 부분 성공으로 처리된다 (Priority: P3)

일부 이슈의 카드 생성이 실패하더라도(스키마 위반, 필수 필드 누락 등) 성공한 이슈들은 그대로 저장되고 파이프라인은 `partial` 상태로 완료된다. 실패 정보는 운영 로그에 기록되어 운영자가 원인을 파악할 수 있다.

**Why this priority**: 단일 이슈 실패가 전체 파이프라인을 중단시키면 운영 부담이 크다. 부분 성공 처리가 서비스 안정성을 보장한다.

**Independent Test**: 일부 기사 그룹에서 스키마 미준수 cards[]가 생성되는 상황을 시뮬레이션하면, 해당 이슈만 스킵되고 나머지 이슈는 정상 저장되며 오류 내역이 로그에 남는다.

**Acceptance Scenarios**:

1. **Given** 특정 이슈에 대해 스키마를 위반한 cards[]가 생성될 때, **When** 검증이 실행되면, **Then** 해당 이슈는 저장되지 않고 오류가 로그에 기록되며, 나머지 유효한 이슈는 정상 저장된다.
2. **Given** 수집된 기사가 0건일 때, **When** 생성 모듈이 실행되면, **Then** AI 호출 없이 즉시 종료되고 이슈 0개로 파이프라인이 완료된다.

---

### Edge Cases

- 수집된 기사가 0건일 때: AI 호출 없이 즉시 종료, 이슈 0개 반환
- 모든 이슈에서 cards 생성이 실패할 때: 파이프라인은 `failed` 상태로 기록
- 동일 종목에 대한 기사가 대량 수집될 때: 동일 entity로 묶여 이슈 1개로 생성(중복 이슈 방지)
- cards[]가 3장 미만이거나 7장 초과일 때: 스키마 검증 실패로 처리, 해당 이슈 스킵
- visual 색상 필드에 hex가 아닌 값이 포함될 때: 스키마 검증 실패, 해당 이슈 스킵
- reason/bullish/bearish 카드에 sources가 없을 때: 스키마 검증 실패, 해당 이슈 스킵
- 생성된 카드에 투자 유도 표현이 포함된 경우: Admin 검토에서 운영자가 반려 처리

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 수집된 기사 목록을 입력받아 주제별로 묶어 이슈 단위로 그룹화한 뒤, 각 이슈에 대해 사전 정의된 cards[] 스키마를 따르는 카드 배열 초안을 생성해야 한다.
- **FR-002**: 생성된 cards[]는 반드시 3~7장 범위여야 하고, 첫 번째 카드는 `cover` 타입, 마지막 카드는 `source` 타입이어야 한다.
- **FR-003**: `visual` 필드의 모든 색상값은 유효한 hex 코드(`#RGB` 또는 `#RRGGBB`) 형식이어야 하며, Tailwind 클래스 문자열은 허용되지 않는다.
- **FR-004**: `reason`, `bullish`, `bearish` 타입 카드는 최소 1개의 출처(sources) 항목을 포함해야 한다.
- **FR-005**: 생성된 카드 내용에는 투자 유도 표현(특정 종목 매수/매도 권유 등)이 포함되어서는 안 된다.
- **FR-006**: 코스피·나스닥·USD/KRW 등 시장 맥락 지표는 일반 종목 이슈와 별도 입력으로 처리되어, 지수 또는 환율 분류의 독립 이슈 초안으로 생성된다.
- **FR-007**: 특정 이슈의 카드 생성이 스키마 검증 실패로 저장 불가 상태가 되더라도, 다른 이슈 처리를 중단하지 않는다.
- **FR-008**: 카드 생성 과정에서 발생한 모든 오류(이슈명, 오류 내용)는 파이프라인 로그에 기록된다.
- **FR-009**: 수집된 기사가 0건인 경우, AI 호출 없이 즉시 빈 결과(이슈 0개)를 반환한다.
- **FR-010**: 생성된 이슈 초안에는 카드 배열 외에도 종목 식별자(entity_id, entity_name, entity_type), 이슈 제목(title), 변동값(change_value) 필드가 포함되어야 한다.

### Key Entities

- **CollectedArticle**: 수집 모듈이 제공하는 기사 단위. 제목·요약·본문·URL·매체명을 포함한다.
- **GeneratedIssueDraft**: 생성 모듈의 출력 단위. 종목 식별자(entity_id, entity_name, entity_type), 이슈 제목(title), 변동값(change_value), 카드 배열(cards[]), 채널(channel)을 포함한다.
- **Card**: 단일 카드 슬라이드. 타입(cover/reason/bullish/bearish/community/stats/source)으로 분류되며, 타입별로 필수 필드가 다르다.
- **CardVisual**: 카드의 배경 그라디언트(`bg_from`, `bg_via`, `bg_to`)와 강조색(`accent`)을 정의하는 색상 집합. 모두 hex 코드여야 한다.
- **CardSource**: 카드에 인용된 기사 출처. 제목(title), URL(url), 도메인(domain)을 포함한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 수집된 기사가 존재하는 파이프라인 실행 중 90% 이상에서 1개 이상의 유효한 이슈 초안이 생성된다.
- **SC-002**: 스키마 검증을 통과한 이슈 초안의 cards[]는 100% cover 시작·source 종료·3~7장 범위를 준수한다.
- **SC-003**: 스키마 검증을 통과한 이슈 초안의 모든 visual 필드에 hex 위반 색상이 존재하지 않는다.
- **SC-004**: reason/bullish/bearish 카드에서 sources 누락으로 인한 검증 실패 비율이 전체 생성 시도의 10% 미만이다.
- **SC-005**: 부분 성공 시나리오에서 유효한 이슈들은 항상 저장된다 (단일 이슈 실패가 다른 이슈 저장을 막지 않음).
- **SC-006**: 수집 기사 0건 입력 시 AI 호출이 발생하지 않아 불필요한 처리 비용이 발생하지 않는다.
- **SC-007**: 운영자 검토 결과 생성된 카드에서 투자 권유 표현이 검출되는 비율이 전체 카드의 5% 미만이다.

## Assumptions

- 맥락 카드(코스피·나스닥·USD/KRW) 생성에 필요한 당일 지수 데이터 입력 방식은 구현 단계에서 결정한다. MVP에서는 파이프라인 호출 시 별도 경로로 전달한다고 가정한다.
- `channel` 필드는 MVP에서 단일 값(`'v1'`)으로 고정되며, 다중 채널 관리는 MVP 범위 밖이다.
- 기사 그룹화(종목별 기사 묶기)는 AI가 입력 기사를 분석하여 자동으로 수행한다. 사전 종목 목록 주입 없이 기사 내용으로부터 entity를 도출한다.
- 스키마 검증에 실패한 이슈는 DB에 저장하지 않는다(부분 완성 초안 비저장).
- 이 모듈의 출력은 항상 Admin 검토를 거쳐 발행된다. 자동 발행은 없다.
- 카드 스키마(타입 목록, 필드 구조)는 SRS § 4.2에서 정의한 구조로 고정되며, 이 모듈은 구조를 변경하지 않고 데이터만 채운다.
