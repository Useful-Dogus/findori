## User Scenarios & Testing *(mandatory)*

### User Story 1 - 피드 발행 (Priority: P1)

운영자가 날짜별 피드 검토 화면에서 이슈들을 확인한 후, 발행 버튼을 클릭하여 해당 날짜의 피드를 공개 상태로 전환한다. 발행 즉시 공개 피드에서 해당 날짜의 승인된 이슈들이 노출된다.

**Why this priority**: 파이프라인과 이슈 검토 기능이 완성되어 있음에도 발행 수단이 없으면 어떤 콘텐츠도 사용자에게 노출할 수 없다. MVP의 핵심 게이팅 기능이다.

**Independent Test**: 날짜별 피드 검토 화면에서 승인된 이슈가 1건 이상인 draft 피드에 대해 발행 버튼을 클릭하고, 즉시 해당 피드의 상태가 published로 바뀌며 공개 피드 API에서 조회 가능한지 확인한다.

**Acceptance Scenarios**:

1. **Given** draft 상태의 피드에 승인된 이슈가 1건 이상 있을 때, **When** 운영자가 발행 버튼을 클릭하면, **Then** 피드 상태가 즉시 published로 전환되고 발행 일시가 기록된다.
2. **Given** 발행이 완료된 직후, **When** 운영자가 피드 검토 화면을 보면, **Then** 발행 완료 상태와 발행 일시가 표시된다.
3. **Given** 발행이 완료된 직후, **When** 공개 피드에서 해당 날짜를 조회하면, **Then** 승인된 이슈들이 노출된다 (반려된 이슈는 제외).

---

### User Story 2 - 발행 차단: 승인 이슈 없음 (Priority: P2)

운영자가 아직 이슈 검토를 완료하지 않은 상태(승인 이슈 0건)에서 실수로 발행을 시도했을 때, 시스템이 발행을 차단하고 원인을 안내한다.

**Why this priority**: 잘못된 발행을 방지하여 빈 피드가 공개되는 상황을 막는다. 운영 실수를 사전에 차단하는 가드레일 역할이다.

**Independent Test**: 승인된 이슈가 없는 draft 피드에서 발행을 시도하면 발행이 거부되고 안내 메시지가 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** draft 상태의 피드에 승인된 이슈가 0건일 때, **When** 운영자가 발행을 시도하면, **Then** 발행이 거부되고 "승인된 이슈가 없어 발행할 수 없습니다" 취지의 안내가 표시된다.
2. **Given** 발행이 거부된 후, **When** 운영자가 피드 검토 화면을 보면, **Then** 피드 상태는 draft로 유지된다.

---

### User Story 3 - 이미 발행된 피드 (Priority: P3)

운영자가 이미 published 상태인 피드를 재발행하려는 상황에서, 시스템이 중복 발행을 방지한다.

**Why this priority**: 데이터 무결성 보호와 운영 혼란 방지를 위한 방어적 요구사항이다.

**Independent Test**: published 상태의 피드 검토 화면에서 발행 버튼이 비활성화 또는 비노출되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 이미 published 상태인 피드일 때, **When** 운영자가 해당 피드 검토 화면을 열면, **Then** 발행 버튼이 비활성화되거나 표시되지 않는다.
2. **Given** 이미 published 상태인 피드에 직접 발행 요청을 보냈을 때, **When** 시스템이 요청을 처리하면, **Then** 오류 응답을 반환하고 피드 상태는 변경되지 않는다.

---

### Edge Cases

- 피드에 이슈가 전혀 없는 경우(0건) 발행 시도 시 차단되어야 한다.
- 모든 이슈가 반려(rejected) 상태인 경우 발행 시도 시 차단되어야 한다.
- 발행 도중 네트워크 오류가 발생하면 피드 상태는 draft로 유지되어야 한다 (부분 발행 없음).
- 존재하지 않는 날짜의 피드에 발행 요청이 들어오면 404 오류를 반환해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 draft 상태 피드의 검토 화면에 발행 버튼을 표시해야 한다.
- **FR-002**: 시스템은 발행 요청 시점에 해당 피드에 승인된 이슈(approved)가 1건 이상 존재하는지 검증해야 한다.
- **FR-003**: 시스템은 검증 통과 시 피드의 상태를 published로, 발행 일시를 현재 시각으로 원자적으로 업데이트해야 한다.
- **FR-004**: 시스템은 발행 성공 후 운영자 화면에 변경된 상태(published)와 발행 일시를 즉시 반영해야 한다.
- **FR-005**: 시스템은 승인된 이슈가 없을 경우 발행 요청을 거부하고 원인을 명확한 메시지로 안내해야 한다.
- **FR-006**: 시스템은 이미 published 상태인 피드에 대한 발행 요청을 거부해야 한다.
- **FR-007**: 발행된 피드의 공개 조회 시 반려(rejected) 이슈는 제외되고 승인(approved) 이슈만 노출되어야 한다 (이 필터링은 기존 공개 API 레이어에서 처리하며, 발행 워크플로우는 피드 상태 전환에만 집중한다).
- **FR-008**: 운영자 인증이 유효하지 않으면 발행 요청은 거부되어야 한다.

### Key Entities

- **Feed (피드)**: 특정 날짜의 이슈 카드 스트림. `draft` 또는 `published` 상태를 가지며, published 전환 시 발행 일시가 기록된다.
- **Issue (이슈)**: 피드에 속한 개별 이슈 카드. `draft`, `approved`, `rejected` 상태를 가진다. 발행 워크플로우에서 approved 이슈의 존재 여부가 발행 가능 조건의 기준이 된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 운영자가 이슈 검토 완료 후 피드 발행까지 걸리는 시간이 30초 이내이다.
- **SC-002**: 발행 버튼 클릭 후 피드 상태 전환과 화면 갱신이 3초 이내에 완료된다.
- **SC-003**: 승인된 이슈가 없는 피드에 대한 발행 시도가 100% 차단된다.
- **SC-004**: 발행 완료 후 공개 피드에서 해당 날짜 이슈가 즉시 조회 가능하다 (별도 조작 불필요).
- **SC-005**: 발행된 피드의 반려 이슈는 공개 피드에서 0건 노출된다.

## Assumptions

- 발행은 되돌릴 수 없다(published → draft 롤백 없음). MVP 범위에서 발행 취소는 Out of Scope이다.
- 발행 조건은 "승인된 이슈 1건 이상"이며, draft 이슈가 섞여 있어도 발행 가능하다.
- 발행 대상 이슈 필터링(approved만 공개)은 기존 공개 API 레이어에서 처리하므로 이 기능은 피드 상태 전환에만 집중한다.
- 운영자 인증은 기존 Admin 세션 체계를 그대로 사용한다.

## Out of Scope

- 발행 취소(published → draft 롤백)
- 예약 발행(특정 시각에 자동 발행)
- 발행 알림(Slack, 이메일 등 외부 알림)
- 복수 운영자 동시 발행 충돌 처리
