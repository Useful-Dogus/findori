# Feature Specification: 에이전트 산출물 작성 규칙 통합

**Feature Branch**: `042-agent-artifact-conventions`
**Created**: 2026-03-03
**Status**: Draft
**Input**: User description: "speckit.specify 에이전트 산출물 작성 규칙 통합 #42 (https://github.com/Useful-Dogus/findori/issues/42)"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 동일 작업에서 에이전트 무관 산출물 일관성 확보 (Priority: P1)

개발자가 Claude Code와 Codex를 번갈아 사용하더라도, 스펙·GitHub 이슈·커밋 메시지·PR 본문이 동일한 형식 규칙을 따른다. 한 에이전트가 만든 이슈를 다른 에이전트가 이어받아 작업할 때 형식 불일치로 인한 혼란이 없다.

**Why this priority**: 형식 불일치는 커밋 히스토리 가독성을 낮추고 스펙·이슈·브랜치 번호 추적 비용을 증가시킨다. 이 문제를 해결하는 것이 이 피처의 핵심이다.

**Independent Test**: 동일한 작업 설명으로 Claude Code와 Codex에게 이슈 초안을 작성하게 한 뒤, 두 초안의 필수 섹션 목록이 일치하는지 확인하여 단독으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 공통 정책 문서가 저장소에 적용된 상태에서, **When** Claude Code와 Codex 각각이 동일 작업으로 GitHub 이슈 초안을 작성하면, **Then** 두 초안 모두 동일한 필수 섹션(배경, 범위, 완료 기준 등)을 포함한다.
2. **Given** 개발자가 커밋을 생성할 때, **When** 에이전트가 커밋 메시지를 제안하면, **Then** 허용된 Conventional Commits 타입 집합만 사용한 메시지가 생성된다.
3. **Given** PR을 생성할 때, **When** 에이전트가 PR 본문을 작성하면, **Then** 연결된 스펙 경로와 이슈 번호가 올바른 키워드(`Closes` 또는 `Part of`)와 함께 포함된다.

---

### User Story 2 - 스펙·브랜치·이슈 번호 정렬 규칙 준수 (Priority: P1)

스펙 디렉터리 번호, 브랜치 번호, GitHub 이슈 번호가 항상 일치한다. 어떤 에이전트가 브랜치를 생성하거나 이슈를 참조하더라도 번호 체계가 어긋나지 않는다.

**Why this priority**: 번호 불일치는 추적 비용을 높이고 "어떤 이슈가 어떤 브랜치에서 구현 중인가"를 파악하는 데 불필요한 검색을 요구한다.

**Independent Test**: `specs/NNN-*`, `git branch NNN-*`, GitHub 이슈 `#NNN` 세 곳의 번호가 일치하는지 확인하여 단독으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 새 피처가 GitHub 이슈 #42로 등록된 상태에서, **When** 에이전트가 브랜치와 스펙 디렉터리를 생성하면, **Then** 브랜치명은 `042-*`, 스펙 경로는 `specs/042-*` 형태를 따른다.
2. **Given** 공통 정책 문서에 번호 정렬 규칙이 명시된 상태에서, **When** 에이전트가 스펙 디렉터리를 생성하면, **Then** GitHub 이슈 번호와 3자리 패딩으로 맞춘 번호를 사용한다.

---

### User Story 3 - 공통 코드 품질 기준 에이전트 공유 (Priority: P2)

가독성, 예측 가능성, 응집도, 결합도, 기본 성능 규칙 등 코드 품질 기준이 단일 정책 문서에 정의되어 있어, Claude Code와 Codex 모두 동일한 품질 기준으로 코드를 생성한다.

**Why this priority**: 산출물 형식이 일관되더라도 코드 품질 기준이 에이전트마다 다르면 리뷰 비용이 증가하고 코드베이스 일관성이 저하된다.

**Independent Test**: 동일한 기능 구현 요청으로 두 에이전트가 생성한 코드가 모두 공통 품질 기준(예: 단일 책임, 명시적 오류 처리 금지 예외 없음 등)을 충족하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 공통 정책 문서에 코드 품질 기준이 정의된 상태에서, **When** 에이전트가 코드를 생성하면, **Then** 정의된 가독성·응집도·결합도 기준을 벗어나지 않는다.
2. **Given** 에이전트가 수정을 제안할 때, **When** 기존 코드가 공통 품질 기준을 위반한 경우, **Then** 에이전트는 해당 기준을 명시하며 수정을 제안한다.

---

### User Story 4 - hotfix 예외 흐름 적용 (Priority: P3)

긴급 hotfix 상황에서 에이전트가 표준 워크플로우 일부를 생략할 수 있음을 알고, 최소 필수 항목(커밋 타입, 이슈 참조 등)만 준수하며 빠르게 산출물을 생성한다.

**Why this priority**: 일반 규칙이 엄격할수록 긴급 상황에서 유연성이 필요하다. hotfix 예외 정의 없이는 에이전트가 긴급 수정 시 혼란을 겪거나 불필요한 단계를 수행할 수 있다.

**Independent Test**: hotfix 상황 설명과 함께 에이전트에 커밋 메시지 작성을 요청했을 때, 에이전트가 `hotfix:` 타입을 사용하고 정의된 최소 항목만 포함하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** hotfix 예외 흐름이 정책 문서에 정의된 상태에서, **When** 에이전트가 긴급 수정 커밋을 생성하면, **Then** `hotfix:` 타입과 최소 필수 항목(이슈 참조 등)을 포함한다.
2. **Given** hotfix 브랜치에서 작업 중일 때, **When** PR을 생성하면, **Then** 스펙 경로 없이도 이슈 번호 참조만으로 PR 본문이 유효하다고 판단된다.

---

### Edge Cases

- 이슈 번호가 없는 사내 작업(리팩터링, 문서 수정 등) 커밋에는 어떤 번호·타입을 사용하는가?
- 공통 정책 문서와 에이전트 루트 문서(CLAUDE.md, AGENTS.md)의 규칙이 충돌하는 경우 어느 쪽이 우선하는가?
- 기존 커밋/PR이 새 규칙을 위반하더라도 소급 적용하지 않음을 에이전트가 인지하고 있는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 저장소 내에 산출물 작성 규칙과 코드 품질 기준을 포함하는 공통 정책 문서 1개가 존재해야 한다.
- **FR-002**: 공통 정책 문서는 스펙, GitHub 이슈, 커밋 메시지, PR 각각의 제목 규칙과 필수 섹션을 정의해야 한다.
- **FR-003**: 공통 정책 문서는 스펙 디렉터리·브랜치·이슈 번호 정렬 규칙을 명시해야 한다.
- **FR-004**: 공통 정책 문서는 허용된 Conventional Commits 타입 집합을 확정하고 나열해야 한다.
- **FR-005**: 공통 정책 문서는 PR 내 `Closes` vs `Part of` 링크 키워드 사용 규칙을 정의해야 한다.
- **FR-006**: 공통 정책 문서는 코드 품질 공통 기준(가독성, 예측 가능성, 응집도, 결합도, 기본 성능 규칙)을 포함해야 한다.
- **FR-007**: 공통 정책 문서는 hotfix 예외 흐름과 최소 필수 항목을 정의해야 한다.
- **FR-008**: Claude Code의 에이전트 지침 문서(CLAUDE.md 또는 연결된 공통 지침 파일)가 공통 정책 문서를 참조하도록 갱신되어야 한다.
- **FR-009**: Codex의 에이전트 지침 문서(AGENTS.md 또는 연결된 공통 지침 파일)가 공통 정책 문서를 참조하도록 갱신되어야 한다.
- **FR-010**: 에이전트 루트 문서는 공통 정책 문서의 규칙을 중복 기술하지 않고 참조로 대체해야 한다.

### Key Entities

- **공통 정책 문서**: 에이전트 종류와 무관하게 적용되는 산출물 작성 규칙(스펙·이슈·커밋·PR 형식), 번호 정렬 규칙, 코드 품질 기준, hotfix 예외 흐름을 담는 단일 문서
- **산출물 규칙**: 스펙 문서, GitHub 이슈, 커밋 메시지, PR 본문 각각의 제목 형식·필수 섹션·금지 항목을 정의한 규칙 집합
- **번호 정렬 체계**: 스펙 디렉터리 번호(`NNN-*`), 브랜치 번호(`NNN-*`), GitHub 이슈 번호(`#NNN`)를 3자리 패딩으로 일치시키는 규칙
- **Conventional Commits 타입 집합**: 프로젝트에서 허용하는 커밋 타입 목록 (예: `feat`, `fix`, `docs`, `chore`, `hotfix` 등)
- **에이전트 루트 문서**: CLAUDE.md(Claude Code 진입점), AGENTS.md(Codex 진입점) — 공통 정책 문서를 참조하고 에이전트 전용 설정만 직접 포함

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 동일한 작업 설명으로 두 에이전트(Claude Code, Codex)가 생성한 이슈 초안의 필수 섹션이 100% 일치한다.
- **SC-002**: 공통 정책 문서 적용 후 생성되는 모든 신규 커밋이 허용된 Conventional Commits 타입 집합에서만 타입을 선택한다.
- **SC-003**: 공통 정책 문서 적용 후 생성되는 모든 신규 PR 본문이 스펙 경로 또는 이슈 번호를 포함한다 (hotfix 제외 시 두 항목 모두 포함).
- **SC-004**: 스펙 디렉터리 번호, 브랜치 번호, GitHub 이슈 번호의 불일치 건수가 공통 정책 적용 후 0건으로 유지된다.
- **SC-005**: 공통 정책 문서 내용을 변경할 때 수정이 필요한 파일 수가 1개(공통 정책 문서 단독)로 유지된다.

## Assumptions

- Claude Code는 `CLAUDE.md`(또는 `@`-import로 연결된 파일)를 자동으로 읽고, Codex는 `AGENTS.md`(또는 연결된 파일)를 자동으로 읽는다.
- 에이전트 지침 파일에서 다른 문서를 참조(링크)하는 방식은 이미 Issue #36(에이전트 지침 통합 관리)에서 구축된 구조를 따른다.
- 기존 머지된 커밋·PR은 소급 수정 대상이 아니며, 새로 생성되는 산출물에만 적용된다.
- CI/CD를 통한 규칙 자동 강제(lint 도구 등)는 이 피처의 범위에 포함되지 않으며, 에이전트 지침 기반의 소프트 강제만을 다룬다.
- Conventional Commits 타입 집합은 현재 저장소 이력에서 사용 중인 타입을 기반으로 확정하며, 필요 시 `hotfix` 타입을 추가한다.
