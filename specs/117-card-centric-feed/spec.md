# Feature Specification: 카드 중심 피드 리디자인

**Feature Branch**: `117-card-centric-feed`
**Created**: 2026-04-03
**Status**: Draft
**Input**: User description: "#111 카드 중심 피드 리디자인 이슈를 해결하기 위한 스펙을 작성해주세요."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 카드가 화면을 지배하는 레이아웃 경험 (Priority: P1)

독자가 피드를 열면 텍스트 블록 대신 카드 스택이 화면의 주인공으로 보인다. 현재처럼 제목이 카드와 동등한 크기로 위아래를 차지하지 않고, 카드가 시각적으로 압도적인 비중을 갖는다. 제목과 엔티티명은 카드 아래에 캡션처럼 작게 따라붙는다.

**Why this priority**: 피드의 핵심 가치는 카드 콘텐츠다. 현재 레이아웃은 제목·메타 텍스트가 카드와 같은 비중을 차지해 "읽는 기사"처럼 보인다. 카드를 앞세워야 "보는 뉴스"라는 제품 정체성이 드러난다.

**Independent Test**: 피드 페이지를 열었을 때 각 이슈의 카드 스택 영역이 제목·메타 텍스트 영역보다 시각적으로 현저히 크게 표시되는지 확인. 제목이 카드 아래에 캡션 형태로 위치하는지 확인.

**Acceptance Scenarios**:

1. **Given** 공개 피드 페이지를 열었을 때, **When** 이슈 목록을 스크롤하면, **Then** 각 이슈에서 카드 스택이 화면 대부분을 차지하고 제목·엔티티명은 카드 아래에 작게 표시된다.
2. **Given** 이슈 섹션을 볼 때, **When** 카드 스택과 텍스트 영역의 비중을 비교하면, **Then** 카드 스택이 이슈 섹션 시각 면적의 80% 이상을 차지한다.
3. **Given** 모바일(375px) 화면에서 피드를 볼 때, **When** 첫 번째 이슈를 확인하면, **Then** 카드 스택이 뷰포트 너비의 90% 이상을 채우며 표시된다.

---

### User Story 2 - 캡션 영역의 정보 계층 정리 (Priority: P2)

카드 아래 캡션 영역에서 독자가 "이게 무슨 이슈인지"를 빠르게 파악할 수 있다. **캡션은 카드의 보조 역할**이며, 카드 위아래로 흩어진 메타정보(엔티티 타입 배지, 변화율, 제목, 태그)를 캡션 한 곳에 작게 모아 시선이 카드에서 분산되지 않게 한다. 캡션 정보가 카드보다 강조되어서는 안 된다.

**Why this priority**: 카드 레이아웃 변경(P1)이 완료된 뒤 캡션 정보가 정돈되어야 완성도 있는 경험이 된다. P1 단독으로도 의미 있는 개선이지만, 이 스토리가 더해져야 정보 흐름이 완성된다. 목표는 캡션 정보의 강조가 아니라 **흩어진 정보의 축소·통합**이다.

**Independent Test**: 각 이슈의 캡션 영역에서 엔티티명, 제목, 핵심 메타(변화율 또는 타입)가 단일 영역에 작고 가볍게 표시되는지 확인. 카드 스택 상단에 별도 메타 영역이 없는지 확인. 캡션이 카드보다 시각적으로 두드러지지 않는지 확인.

**Acceptance Scenarios**:

1. **Given** 이슈 카드를 볼 때, **When** 캡션 영역을 확인하면, **Then** 엔티티명(소), 제목(중), 변화율 또는 섹터 태그(소)가 캡션에 작고 가볍게 표시되며, 카드 스택보다 시각적으로 덜 강조된다.
2. **Given** 카드 스택 상단을 확인할 때, **When** 메타 정보 위치를 확인하면, **Then** 엔티티 타입 배지나 변화율이 카드 상단 별도 영역이 아닌 캡션 영역에만 존재한다.
3. **Given** 공유 버튼이 있을 때, **When** 캡션 영역을 확인하면, **Then** 공유 버튼은 캡션 우측에 위치해 캡션 영역 안에서 접근 가능하다.

### User Story 3 - 이슈 간 스크롤 흐름 개선 (Priority: P2)

독자가 여러 이슈를 스크롤할 때 이슈 간 구분이 명확하면서도 카드에서 다음 카드로 시선이 자연스럽게 이어진다. 현재처럼 이슈 섹션 박스(border + rounded + backdrop-blur)가 "기사 블록"처럼 닫혀있지 않고, 카드들이 피드처럼 흐른다.

**Why this priority**: 레이아웃과 캡션이 완성된 뒤 이슈 간 여백과 구분 방식을 다듬어야 전체 피드가 카드 피드 느낌으로 완성된다.

**Independent Test**: 두 이슈 이상이 있는 피드를 스크롤할 때 이슈 간 구분이 시각적으로 명확하면서도 "카드 피드" 느낌이 드는지 확인.

**Acceptance Scenarios**:

1. **Given** 여러 이슈가 있는 피드를 스크롤할 때, **When** 이슈 간 경계를 확인하면, **Then** 두꺼운 border 박스 대신 충분한 수직 여백만으로 이슈가 구분된다 (구분선 없음).
2. **Given** 피드를 위에서 아래로 스크롤할 때, **When** 이슈가 바뀌는 시점을 확인하면, **Then** 카드 스택이 다음 이슈의 카드 스택으로 자연스럽게 이어지는 느낌이다.

---

### Edge Cases

- 이슈가 1개일 때: 단일 이슈도 동일한 카드 중심 레이아웃이 적용되는가?
- 제목이 매우 길 때(40자 이상): 캡션 영역에서 제목이 잘리거나 레이아웃이 깨지지 않는가?
- 변화율이 없는 이슈(change_value 없음): 캡션 영역에서 빈 자리가 어색하게 남지 않는가?
- 데스크톱 화면(1200px 이상): 카드 중심 레이아웃이 와이드 화면에서도 균형 있게 보이는가?
- Market Context 섹션: 리디자인 이후 시장 맥락 카드 섹션과 이슈 섹션의 시각적 일관성이 유지되는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 각 이슈에서 카드 스택은 이슈 섹션 내 시각적으로 가장 큰 영역을 차지해야 한다.
- **FR-002**: 이슈 제목과 엔티티명은 카드 스택 아래 캡션 영역에 표시되어야 한다 (카드 위에 제목 없음).
- **FR-003**: 카드 스택 상단의 별도 메타 정보 영역(엔티티 타입 배지, 변화율)은 제거하고 캡션 영역에 작게 통합되어야 한다. 캡션은 카드의 보조이므로 시각적으로 카드보다 약하게 표현되어야 한다.
- **FR-004**: 공유 버튼은 캡션 영역 내 우측 상단에 위치해야 한다(엔티티명·제목 텍스트와 flex justify-between 구조). 레이아웃이 깨질 경우 캡션 하단 별도 행에 배치하는 것도 허용된다.
- **FR-005**: 이슈를 감싸는 컨테이너 박스(border, rounded, backdrop-blur)는 완전히 제거되어야 한다. 이슈 간 구분은 카드 스택 + 캡션 영역 사이의 여백으로만 처리한다.
- **FR-006**: 카드 스택의 가로 폭은 모바일에서 뷰포트 너비의 90% 이상을 활용해야 한다. 데스크톱에서는 단일 컬럼 + 최대 폭 제한(max-width) 구조로, 모바일보다 더 크고 읽기 편한 카드 및 텍스트 크기를 제공해야 한다 (나이대 높은 사용자·노안 배려).
- **FR-009**: 피드 레이아웃은 모바일(375px)과 데스크톱(1200px 이상) 양쪽에서 모두 최적화되어야 한다. 데스크톱은 더 큰 카드·더 큰 기본 글씨체를 사용해 가독성을 높인다.
- **FR-007**: 기존 카드 슬라이드 동작(스와이프, 진행률 표시, 이전/다음 버튼)은 그대로 유지되어야 한다.
- **FR-008**: 기존 카드 타입별 렌더링 로직(CardBody)은 변경 없이 유지되어야 한다.

### Key Entities

- **이슈 섹션(Issue Section)**: 하나의 이슈를 구성하는 카드 스택 + 캡션 영역의 묶음
- **캡션 영역(Caption Area)**: 카드 스택 아래에 위치하며 엔티티명, 제목, 변화율/태그, 공유 버튼을 포함하는 영역

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 이슈 섹션에서 카드 스택 영역의 시각 면적이 전체 섹션의 80% 이상이다.
- **SC-002**: 제목 텍스트가 카드 스택 하단(아래)에만 위치한다 — 카드 위에 제목 텍스트가 없다.
- **SC-003**: 모바일(375px) 기준으로 카드 스택 가로 폭이 뷰포트의 90% 이상이다. 데스크톱(1200px)에서는 단일 컬럼 + max-width 제한으로 카드가 화면 중앙에 적절한 크기로 표시된다.
- **SC-004**: 기존 카드 슬라이드 인터랙션(스와이프, 버튼 탐색) 동작이 100% 유지된다.
- **SC-005**: 기존 테스트 스위트(vitest)가 전부 통과한다.

## Clarifications

### Session 2026-04-03

- Q: 이슈 섹션 컨테이너 처리 방식 — 컨테이너 유지(스타일만 변경) vs 완전 제거 vs full-bleed → A: 컨테이너 완전 제거, 카드 스택이 배경 위에 직접 노출
- Q: 데스크톱 레이아웃 구조 → A: 단일 컬럼 + 최대 폭 제한. 단, 480px은 너무 좁음. 모바일·PC 모두 최적화 필요. PC는 더 큰 카드·더 큰 글씨 (노안 배려). 모바일: 젊은 사용자, PC: 나이대 높은 사용자 타겟.
- Q: 공유 버튼 캡션 내 배치 방식 → A: 캡션 우측 상단 고정(flex justify-between). 단, 레이아웃이 깨질 경우 캡션 하단 별도 행 배치도 허용.
- Q: 이슈 간 구분 방식 — 여백만 vs 얇은 구분선 → A: 여백(padding/margin)만으로 구분, 선 없음.
- Q: 카드 aspect ratio 변경 여부 → A: 현재 `aspect-[4/5]` 그대로 유지. 비율 변경은 이 스펙 범위 외.

## Assumptions

- Market Context(시장 맥락 지표) 섹션은 이 스펙의 범위에 포함되지 않는다. 기존 표시 방식 유지.
- 카드 내부 콘텐츠(CardBody) 렌더링 로직은 변경 없음 — 순수 레이아웃 변경만 해당.
- 데스크톱에서는 단일 컬럼 + max-width 제한으로 카드가 화면 중앙에 위치한다. 480px max-width는 너무 좁으므로 더 큰 값을 사용한다. 모바일 사용자는 주로 젊은 층, PC 사용자는 나이대 높은 층(노안 배려)으로 대상이 다르므로 데스크톱에서 텍스트·카드 크기를 더 크게 설정한다.
- 기존 `FeedCardStack.tsx`의 슬라이드 로직(터치 이벤트, 진행률 계산)은 그대로 유지한다.
- 카드 aspect ratio(`aspect-[4/5]`)는 변경하지 않는다. 비율 조정은 별도 이슈로 다룬다.
